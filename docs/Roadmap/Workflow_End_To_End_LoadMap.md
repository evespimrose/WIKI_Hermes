# End-to-End 워크플로우 로드맵

## 1. 기반 지식 요약

### 1.1 LLM 컨텍스트 5대 문제
- **컴팩션 절벽**: 압축 시 컨텍스트가 소실돼 매번 처음부터 설명
- **환각 누적**: 세션이 길수록 규칙 우회·존재하지 않는 API 호출 증가
- **토큰 비용 폭발**: 작은 수정 위해 큰 파일 통째로 읽는 낭비
- **세션 단절**: 새 세션시 컨텍스트 0에서 시작, 배경 다시 설명 필요
- **에이전트 디스패치 혼란**: 어떤 작업에 어떤 에이전트 호출할지 복잡한 결정

### 1.2 LLM 통제 철학 6대 원칙
- **토큰은 1급 자원**: 모든 행동에 토큰 비용 질문 선행, 필요할 때만 외부에서 가져옴
- **상태는 LLM 밖에**: 영속 상태는 모두 파일로 외부화, "기억"→"읽어서 확인" 패턴
- **프롬프트보다 훅**: 프롬프트는 신뢰 불가, 결정론적 강제는 훅으로 구현
- **모듈 독립 설치**: 4개 독립 모듈로 분해, 단독·결합·부분 설치 모두 허용
- **단일 책임 강제**: 역할의 모호함은 환각 원천, 모든 역할은 단 하나의 책임만
- **규모별 차별화**: 모든 작업에 동일 절차는 낭비, 플랜 존재 여부로 규모 판단

## 2. 궁극적인 목표

> **"LLM의 태생적 한계(비결정론, 컨텍스트 휘발성)를 기계적인 아키텍처와 프로토콜로 완전히 통제하는 완벽한 닫힌 생태계(Closed Ecosystem) 구축"**

### 2.1 로컬 프로젝트 코드 베이스
- asmdef 명시적 통제 클린 아키텍쳐 기반 unity + burst + ecs + vcontainer를 이용한 정량적 객체지향 고효율 개발
- 모든 기능명세를 dic에 저장해 BLK 태그 기반 코드그래프 grep 서치
- 매 세션 시작시마다 LLM 위키 정독후 자동 튜닝

### 2.2 LLM 위키
- 모든 마크다운 및 주변노드를 옵시디언 및 codegraph 기반 마크다운 벡터DB로 연결해 로컬 프로젝트 세션에서 참조하도록 환경 조성
- 웹링크 지식 정제 기반 워크플로우 진화

### 2.3 AI
- 입출력 토큰 및 메인 컨텍스트 윈도우 오염 방지 서브에이전트 위주 작업을 통해 1M윈도우 제한이 무색하게 500,600 턴 이상의 massive 윈도우 0-유실로 제어
- 모든 input(doc-context 마크다운), output(예정: 서브에이전트 위주 워크플로우 적용 시 수행결과 역시 구조화한 conclusion을 마크다운으로 저장해 compact 수행해 맥락 보존 극소텍스트 저장 및 codegraph 노드 등록해 키워드 기반 열람)
- llm 위키, codegraph 코드베이스를 전부 유지하며 "지식 유실로 인한 환각 없는" 유토피아적 개발 환경 구축

## 3. 환각 없는 유토피아적 개발 환경: 비전 분석

### 3.1 로컬 코드베이스: 극단적 제어와 고성능의 결합
- **정량적 객체지향과 DOTS**: asmdef 기반의 단방향 Clean Architecture 레이어 통제 위에, Unity ECS(DOTS)와 Burst 컴파일러, VContainer를 결합하여 제로 얼로케이션(Zero-allocation) 수준의 고효율 로직을 전개하는 것은 아키텍처 설계상 완벽한 궁합입니다.
- **공간 맵핑(Spatial Mapping)**: 모든 기능 명세가 dictionary.md와 // [BLK-XXX] 태그로 관리되며, 무차별 탐색을 배제하고 Codegraph grep으로만 코드에 접근합니다. 매 세션 시작 시 위키를 정독해 자동 튜닝되는 환경은 에이전트가 "어디에 무엇이 있는지" 절대 잊지 않게 만듭니다.

### 3.2 LLM 위키: 수동적 도서관에서 능동적 공유 신경망으로
- **벡터 DB 기반 온디맨드 시맨틱 질의**: 단순한 마크다운 문서 더미를 넘어, Obsidian과 Codegraph 기반의 순수 문서 벡터 DB로 연결하여 에이전트들이 필요할 때만 "온디맨드 시맨틱 질의"를 수행하는 능동적 신경망을 구축합니다.
- **웹 링크 지식 정제 통합**: 사용자 요청 시 웹 링크 지식까지 원자 단위 정제하여 워크플로우에 통합함으로써, 에이전트들이 매 턴마다 불필요한 컨텍스트를 주입받지 않고도 프로젝트의 철학과 의도를 정확히 참조할 수 있게 합니다.

### 3.3 AI 및 컨텍스트 방어: 0-유실, 무한 턴의 실현
- **컨텍스트 오염 방지**: 1M 토큰 윈도우라는 물리적 한계를 3-Tier 에이전트 스튜디오의 서브 에이전트 디스패치로 분산시켜 해결합니다.
- **환각 제로(Zero-Hallucination)와 압축**: 입력은 doc-context 마크다운으로 구조화하고, 서브 에이전트의 작업 결과(Conclusion)는 메인 컨텍스트 윈도우를 침범하지 않고 극도로 구조화 & 압축(Compact)하여 마크다운 및 Codegraph 노드로 영구 보존합니다. 이 4계층 메모리 구조를 통해 500~600턴 이상의 Massive 윈도우에서도 과거의 맥락이 밀려나 환각을 일으키는 일을 원천 차단합니다.

## 4. 현실적인 직언 (Candor & Reality Check)

이 "유토피아"는 이론적으로 매우 우아하고 강력하지만, 실제 운용 시 가장 큰 병목은 서브 에이전트의 출력(Conclusion)을 얼마나 균일하고 압축된 고품질의 마크다운/노드로 파싱해낼 수 있는가에 달려 있습니다. 결과물이 규격화되지 않고 노이즈가 섞인 채로 저장된다면, Vector DB와 Codegraph 인덱스에 오염이 누적되고 이는 결국 검색 정밀도 하락과 미세한 환각으로 이어지게 됩니다. 시스템 전체의 생명선은 압축(Compaction)과 인덱스 갱신의 무결성을 강제하는 훅(Hook) 시스템에 있습니다.

## 5. 궁극적 사용자 UX

"~~ 해줘"라는 컨텍스트 윈도우만 입력하면 자동으로 이 워크플로우를 탈옥없이 사용하는 mastermind 오케스트레이터 에이전트가 반응해 여러 에이전트로 구성된 팀을 만들어 end-to-end까지 클로드의 /goal과 /loop를 이용해 초미세토큰을 이용한 초극효율 개발을 완수합니다.

**에이전트 팀 구성**:
- 팀 오케스트레이터
- prototype
- 적대적 검증자(끊임없이 결과물을 의심하며 "정말로 그러한지" 질문 및 구멍 찾아내 사용자 및 타 에이전트에게 인계)
- 퀄리티 유지
- 기록 갱신 & 편집
- 기획
- 코드베이스 조율
- 배포
- 유지보수
- 디버깅 등

## 6. 마스터마인드 자율 에이전트 팀 아키텍처 매핑

사용자가 추상적인 의도만 던지면, 하네스 인프라가 작동하여 .claude/agents/와 skills/ 내부에 명확한 역할과 입출력 프로토콜을 가진 하위 팀을 동적으로 빌드하거나 라우팅합니다.

### 6.1 아키텍처 다이어그램

```text
         [사용자 의도 입력: "~~ 해줘"] (Intent-Based)
                       │
                       ▼
         ┌──────────────────────────┐
         │ Mastermind Orchestrator  │ ── (PM 역할: 워크플로 감시/조율)
         └──────────────────────────┘
                       │
         ┌─────────────┴─────────────┐ (동적 커뮤니케이션 활성화)
         ▼                           ▼
 ┌───────────────┐           ┌───────────────┐
 │ Prototype     │ ◄───────► │ 적대적 검증자   │ (끊임없는 의심, 구멍 찾기)
 │ (코드/기획)    │           │ (Adversarial)  │
 └───────────────┘           └───────────────┘
         │                           │
         └─────────────┬─────────────┘
                       ▼
 ┌──────────────────────────────────────────┐
 │ 퀄리티 유지 (Quality Sentinel)             │ ── (Clean Arch & 컨벤션 하드 게이트)
 └──────────────────────────────────────────┘
                       │
                       ▼
 ┌──────────────────────────────────────────┐
 │ 기록 갱신 & 편집 (Reporter)                │ ── (결과물을 압축 마크다운으로 추출)
 └──────────────────────────────────────────┘
```

### 6.2 역할 분담과 단일 책임 (Single Responsibility)

- **Mastermind 오케스트레이터**: 세부 구현에는 개입하지 않고, 전체 워크플로가 목표(/goal)와 루프(/loop) 안에서 올바르게 수렴하는지만 감시하는 최상위 관제탑 역할을 수행합니다.
- **Prototype 에이전트 & 적대적 검증자 (Adversarial Verifier)**: ~/.claude/settings.json에서 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAM=1 설정을 통해 서로 격리되지 않고 대화(Cross-talk)합니다. 프로토타입이 초안을 내면, 검증자가 "정말로 예외 처리가 되었는가?", "Burst 컴파일과 Unity ECS 규칙을 준수했는가?"라며 끊임없이 청문회를 열어 품질을 극한으로 끌어올립니다.
- **Quality Sentinel & Reporter**: 검증이 끝난 결과물이 클린 아키텍처 규칙을 통과하는지 물리적으로 검사하고[cite: 4, 10], 최종 수행 결과를 극소 텍스트 형태의 구조화된 결론(Conclusion) 마크다운으로 뽑아내어 Codegraph 노드에 등록합니다.

## 7. 초미세 토큰(Ultra-Micro Tokens) 달성 위한 핵심 현실 점검

하네스 엔지니어링 연구에 따르면, 멀티 에이전트 팀이 서로 대화하며 협업할 경우 토큰 소모량이 평균 30% 이상 증가하는 명확한 트레이드오프가 존재합니다. 사용자가 원하는 "초미세 토큰을 이용한 초극효율 개발"을 달성하려면, 이 에이전트 간의 대화 레이어에 앞서 구축한 통제 섀시를 결합해야만 합니다.

### 7.1 대화 자원의 컨텍스트화 금지 (주방 작업대 원칙)
에이전트들이 서로 대화할 때 파일 전체나 무관한 컨텍스트를 넘겨주면 토큰이 폭발합니다. 에이전트 간 소통은 오직 dictionary.md 기반의 BLK 좌표 시스템과 API 명세(프로토콜)만 주고받도록 스킬(Skill) 수준에서 통제해야 합니다.

### 7.2 루프(/loop)의 강제 컴팩션
적대적 검증자와 프로토타입이 수차례 루프를 돌며 컨텍스트가 쌓이면 즉시 /compact를 실행하되, 컴팩션 직전의 핵심 논리적 분기점과 승인된 명세의 "마크다운 코드그래프 연결용 키워드(질의 트리거용 단어)"만 4계층 메모리(L2/L3)에 남겨 유실률을 0%로 제어합니다.

## 8. 유토피아를 향한 다음 단계

전체 시스템의 가장 큰 병목은 **서브 에이전트의 출력(Conclusion)을 얼마나 균일하고 압축된 고품질의 마크다운/노드로 파싱해낼 수 있는가**에 달려 있습니다. 이를 해결하기 위해 앞으로 다음을 수행해야 합니다.

### 8.1 Conclusion 파싱 표준화
- 서브 에이전트 출력 형식의 강력한 스키마 정의
- 필수 메타데이터(출력 유형, 대상 프로젝트, 생성 시간, 관련 BLK 좌표) 표준화
- 파싱 실패시 fallback 메커니즘 구축

### 8.2 압축(Compaction) 전략 고도화
- 컨텍스트 유실률 0%를 유지하는 압축 알고리즘 개발
- "마크다운 코드그래프 연결용 키워드" 자동 추출 및 보존
- 압축 전/후 검증 단계 강제화

### 8.3 Hook 시스템 구축
- 압축과 인덱스 갱신의 무결성을 강제하는 물리적 Hook 구현
- 결과물 규격화 검증 게이트
- 노이즈가 섞인 입력 자동 차단 메커니즘
