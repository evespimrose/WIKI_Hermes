End-to-End 워크플로우 로드맵

1. 기반 지식 요약

1.1 LLM 컨텍스트 5대 문제

  • 컴팩션 절벽: 압축(/compact) 시 컨텍스트가 소실돼 매번 처음부터 설명
  • 환각 누적: 세션(session)이 길수록 규칙 우회·존재하지 않는 API 호출 증가
  • 토큰 비용 폭발: 작은 수정 위해 큰 파일 통째로 읽는 낭비
  • 세션 단절: 새 세션 시 컨텍스트 0에서 시작(cold-start), 배경 다시 설명 필요
  • 에이전트 디스패치 혼란: 어떤 작업에 어떤 에이전트 호출할지(dispatch) 복잡한 결정

1.2 LLM 통제 철학 6대 원칙

  • 토큰은 1급 자원: 모든 행동에 토큰(token) 비용 질문 선행, 필요할 때만 외부에서 가져옴
  • 상태는 LLM 밖에: 영구 시킬 정보는 모두 파일로 외부화, "기억"→"읽어서 확인" 패턴
  • 프롬프트보다 훅: 프롬프트(prompt)는 신뢰 불가, 결정론적 강제는 훅(hook)으로 구현
  • 모듈 독립 설치: 4개 독립 모듈로 분해, 단독·결합·부분 설치 모두 허용
  • 단일 책임 강제: 역할의 모호함은 환각 원천, 모든 역할은 단 하나의 책임만
  • 규모별 차별화: 모든 작업에 동일 절차는 낭비, 플랜 존재 여부로 규모 판단

2. 궁극적인 목표

"LLM의 태생적 한계(비결정론, 컨텍스트 휘발성)를 기계적인 아키텍처와 프로토콜로 완전히 통제하는 완벽한 닫힌 생태계(Closed Ecosystem) 구축"

2.1 로컬 프로젝트 코드 베이스

  • asmdef(asmdef) 명시적 통제 클린 아키텍쳐(Clean Architecture) 기반 unity(Unity) + burst(Burst) + ecs(ECS) + vcontainer(VContainer)를 이용한 정량적 객체지향 고효율 개발
  • 모든 기능명세를 dic에 저장해 BLK 태그(BLK) 기반 codegraph(Codegraph) grep 서치
  • 매 세션 시작시마다 LLM 위키 정독후 자동 튜닝

2.2 LLM 위키

  • 모든 마크다운 및 주변노드를 옵시디언(Obsidian) 및 codegraph 기반 마크다운 벡터DB(Vector DB)로 연결해 로컬 프로젝트 세션에서 참조하도록 환경 조성
  • 웹링크 지식 정제 기반 워크플로우 진화

2.3 AI

  • 입출력 토큰 및 메인 컨텍스트 윈도우 오염 방지 서브에이전트(Sub-agent) 위주 작업을 통해 1M윈도우(1M token window) 제한이 무색하게 500,600 턴 이상의 massive 윈도우(massive window) 0-유실로 제어
  • 모든 input(doc-context)(doc-context) 마크다운, output(예정: 서브에이전트 위주 워크플로우 적용 시 수행결과 역시 구조화한 conclusion(Conclusion)을 마크다운으로 저장해 compact 수행해 맥락 보존 극소텍스트 저장 및 codegraph 노드 등록해 키워드 기반 열람) 마크다운으로 저장 & codegraph 노드로 등록
  • llm 위키, codegraph 로컬 데이터베이스를 전부 유지하며 "지식 유실로 인한 환각 없는" 유토피아적 개발 환경 구축

3. 환각 없는 유토피아적 개발 환경: 비전 분석

3.1 로컬 코드베이스: 극단적 제어와 고성능의 결합

  • 정량적 객체지향과 DOTS: asmdef 기반의 단방향 Clean Architecture 레이어 통제 위에, Unity ECS(DOTS)와 Burst 컴파일러, VContainer를 결합하여 제로 얼로케이션(Zero-allocation) 수준의 고효율 로직을 전개하는 것은 아키텍처 설계상 완벽한 궁합입니다.
  • 공간 맵핑(Spatial Mapping): 모든 기능 명세가 dictionary.md(dictionary.md)와 // [BLK-XXX] BLK 태그로 관리되며, 무차별 탐색을 배제하고 Codegraph grep으로만 코드에 접근합니다. 매 세션 시작 시 위키를 정독해 자동 튜닝되는 환경은 에이전트가 "어디에 무엇이 있는지" 절대 잊지 않게 만듭니다.

3.2 LLM 위키: 수동적 도서관에서 능동적 공유 신경망으로

  • 벡터 DB 기반 온디맨드 시맨틱 질의: 단순한 마크다운 문서 더미를 넘어, Obsidian과 Codegraph 기반의 벡터 DB로 연결하여 에이전트들이 필요할 때만 "온디맨드 시맨틱 질의(on-demand semantic query)"를 수행하는 능동적 신경망을 구축합니다.
  • 웹 링크 지식 정제 통합: 웹 링크 지식까지 정제하여 워크플로우에 통합함으로써, 에이전트들이 매 턴마다 불필요한 컨텍스트를 주입받지 않고도 프로젝트의 철학과 의도를 정확히 참조할 수 있게 합니다.

3.3 AI 및 컨텍스트 방어: 0-유실, 무한 턴의 실현

  • 컨텍스트 오염 방지: 1M 토큰 윈도우라는 물리적 한계를 3-Tier 에이전트 스튜디오의 서브 에이전트 디스패치로 분산시켜 해결합니다.
  • 환각 제로(Zero-Hallucination)와 압축: 입력은 doc-context 마크다운으로 구조화하고, 서브 에이전트의 작업 결과(Conclusion)는 극도로 압축(Compact)하여 마크다운 및 Codegraph 노드로 영구 보존합니다. 이 4계층 메모리 구조를 통해 500~600턴 이상의 Massive 윈도우에서도 과거의 맥락이 밀려나 환각을 일으키는 일을 원천 차단합니다.

4. 현실적인 직언 (Candor & Reality Check)

이 "유토피아"는 이론적으로 매우 우아하고 강력하지만, 실제 운용 시 가장 큰 병목(bottleneck)은 서브 에이전트의 출력(Conclusion)을 얼마나 균일하고 압축된 고품질의 마크다운/노드로 파싱(parsing)해낼 수 있는가에 달려 있습니다.

결과물이 규격화되지 않고 노이즈(noise)가 섞인 채로 저장된다면, Vector DB와 Codegraph 인덱스에 오염이 누적되고 이는 결국 검색 정밀도 하락과 미세한 환각으로 이어지게 됩니다.

시스템 전체의 생명선(lifeline)은 압축(Compaction)과 인덱스 갱신의 무결성(integrity)을 강제하는 훅(Hook) 시스템에 있습니다.

6. 마스터마인드 자율 에이전트 팀 아키텍처 매핑

사용자가 추상적인 의도만 던지면, 하네스(harness)인프라가 작동하여 .claude/agents/와 skills/ 내부에 명확한 역할과 입출력 프로토콜(protocol)을 가진 하위 팀을 동적 빌드하거나 라우팅(routing)합니다.

6.1 역할 분담과 단일 책임 (Single Responsibility)

  • Mastermind 오케스트레이터: 세부 구현에는 개입하지 않고, 전체 워크플로가 목표(/goal)와 루프(/loop) 안에서 올바르게 수렴(converge)하는지만 감시하는 최상위 관제탑 역할을 수행합니다.
  • Prototype 에이전트 & 적대적 검증자 (Adversarial Verifier): ~/.claude/settings.json에서 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAM=1 설정을 통해 서로 격리되지 않고 대화(Cross-talk)합니다. 프로토타입이 초안을 내면, 검증자가 "정말로 예외 처리가 되었는가?", "Burst 컴파일러와 Unity ECS 규칙을 준수했는가?"라며 끊임없이 청문회를 열어 품질을 극한으로 끌어올립니다.
  • Quality Sentinel & Reporter: 검증이 끝난 결과물이 클린 아키텍처 규칙을 통과하는지 물리적으로 검사하고, 최종 수행 결과를 극소 텍스트 형태의 구조화된 결론(Conclusion) 마크다운으로 뽑아내어 Codegraph 노드에 등록합니다.

7. 초미세 토큰(Ultra-Micro Tokens) 달성을 위한 핵심 현실 점검

하네스 엔지니어링 연구에 따르면, 멀티 에이전트(multi-agent) 팀이 서로 대화하며 협업할 경우 토큰 소모량이 평균 30% 이상 증가하는 명확한 트레이드오프(trade-off)가 존재합니다. 사용자가 원하는 "초미세 토큰을 이용한 초극효율 개발"을 달성하려면, 이 에이전트 간의 대화 레이어에 앞서 구축한 통제 섀시를 결합해야만 합니다.

7.1 대화 자원의 컨텍스트화 금지 (주방 작업대 원칙)

에이전트들이 서로 대화할 때 파일 전체나 무관한 컨텍스트를 넘겨주면 토큰이 폭발합니다. 에이전트 간 소통은 오직 dictionary.md 기반의 BLK 좌표 시스템과 API 명세(프로토콜)만 주고받도록 스킬(skill) 수준에서 통제해야 합니다.

7.2 루프(/loop)의 강제 컴팩션

적대적 검증자와 프로토타입이 수차례 루프를 돌며 컨텍스트가 쌓이면 즉시 /compact를 실행하되, 컴팩션 직전의 핵심 논리적 분기점(logical branch)과 승인된 명세의 "마크다운 코드그래프 연결용 키워드(질의 트리거용 단어)"만 4계층 메모리(4-layer memory)에 남겨 유실률을 0%로 제어합니다.

8. 유토피아를 향한 다음 단계

전체 시스템의 가장 큰 병목은 서브 에이전트의 출력(Conclusion)을 얼마나 균일하고 압축된 고품질의 마크다운/노드로 파싱해낼 수 있는가에 달려 있습니다. 이를 해결하기 위해 앞으로 다음을 수행해야 합니다.

8.1 Conclusion 파싱 표준화

  • 서브 에이전트 출력 형식의 강력한 스키마(schema) 정의
  • 필수 메타데이터(metadata)(출력 유형, 대상 프로젝트, 생성 시간, 관련 BLK 좌표) 표준화
  • 파싱 실패시 fallback 메커니즘(fallback) 구축

8.2 압축(Compaction) 전략 고도화

  • 컨텍스트 유실률 0%를 유지하는 압축 알고리즘(algorithm) 개발
  • "마크다운 코드그래프 연결용 키워드" 자동 추출 및 보존
  • 압축 전/후 검증 단계 강제화

8.3 Hook 시스템 구축

  • 압축과 인덱스 갱신의 무결성을 강제하는 물리적 Hook 구현
  • 결과물 규격화 검증 게이트(gate)
  • 노이즈가 섞인 입력 자동 차단 메커니즘(mechanism)
/compact 명령
Claude의 컨텍스트 윈도우를 압축하는 명령으로:
  • 대화 내용을 핵심 정보로 요약
  • 토큰 사용량을 대폭 감소
  • 하지만 정보 소실 위험이 있음
세션(Session)
AI와의 연속된 대화 단위로:
  • 한 번 열면 계속 대화할 수 있는 단위
  • 닫으면 대화 내용이 초기화됨
  • 새 세션은 처음부터 시작
컨텍스트 0에서 시작(Cold-Start)
새로운 세션에서 아무런 배경 지식 없이 시작하는 것:
  • 이전 대화 내용을 기억하지 못함
  • 프로젝트에 대해 다시 설명해야 함
  • 시간과 토큰이 낭비됨
디스패치(Dispatch)
작업을 적절한 에이전트에게 할당하는 것:
  • 어떤 일을 누가 할지 결정
  • 작업 분배와 조정
  • 효율적인 협업을 위해 필요
토큰(Token)
AI가 처리하는 텍스트의 최소 단위로:
  • 단어, 문자, 기호 등의 조합
  • 토큰 수가 많을수록 비용 증가
  • 토큰 절약이 중요한 이유
프롬프트(Prompt)
AI에게 주는 명령이나 질문 텍스트:
  • AI를 어떻게 사용할지 지시
  • 하지만 AI가 항상 정확히 따르지 않음
  • 신뢰성이 떨어질 수 있음
훅(Hook)
시스템이 자동으로 실행하는 규칙:
  • 특정 조건에서 무조건 실행
  • 프롬프트보다 신뢰성 높음
  • 강제적으로 규칙을 적용
asmdef
Unity의 어셈블리 정의 파일:
  • 프로젝트를 모듈로 분리
  • 컴파일 시간 단축
  • 의존성 관리
클린 아키텍쳐(Clean Architecture)
소프트웨어 설계 원칙:
  • 의존성 규칙 준수
  • 테스트 용이성 향상
  • 유지보수 쉬움
Unity
게임 개발 엔진:
  • 2D/3D 게임 제작
  • 크로스플랫폼 지원
  • 에디터 확장 가능
Burst
Unity의 컴파일러:
  • 고성능 코드 생성
  • ECS와 함께 사용
  • 실행 속도 향상
ECS(Entity Component System)
데이터 중심 설계 패턴:
  • 고성능 게임 개발
  • 데이터 지역성
  • 병렬 처리 용이
VContainer
Unity용 DI 컨테이너:
  • 의존성 주입
  • 테스트 용이
  • 코드 결합도 낮춤
제로 얼로케이션(Zero-allocation)
메모리 할당 없이 실행:
  • 가비지 컬렉션 피함
  • 성능 안정성
  • 프레임 드롭 방지
BLK 태그
프로젝트 구조 태그 시스템:
  • 기능 위치 표시
  • 코드 찾기 쉬움
  • 무차별 탐색 방지
Codegraph
코드 구조 시각화 도구:
  • 코드 관계 보기
  • grep으로 찾기
  • 노드로 관리
dictionary.md
프로젝트 용어 사전:
  • 기능 명세 저장
  • 용어 정의
  • BLK 태그 관리
옵시디언(Obsidian)
노트 기반 지식 관리 도구:
  • 양방향 링크
  • 로컬 저장
  • 확장성 높음
벡터 DB(Vector DB)
의미 기반 검색 데이터베이스:
  • 유사 내용 찾기
  • 시맨틱 검색
  • 지식 연결
서브에이전트(Sub-agent)
특정 작업 담당 AI:
  • 역할 분담
  • 전문성 높임
  • 협업 효율
1M token window
AI가 처리할 수 있는 최대 텍스트 양:
  • 약 1백만 토큰
  • 긴 문서 처리
  • 하지만 제한 있음
massive window
매우 긴 대화 세션:
  • 500~600 턴 이상
  • 지식 유지
  • 환각 방지
doc-context
문서 컨텍스트 주입 스킬:
  • 문서 자동 로드
  • 용어 사전 적용
  • 반복 전송 방지
Conclusion
작업 결과 요약:
  • 결과 압축
  • 영구 저장
  • 나중에 참조
on-demand semantic query
필요할 때만 검색:
  • 의미 기반 찾기
  • 불필요한 컨텍스트 피함
  • 토큰 절약
bottleneck
병목 현상:
  • 전체 시스템 성능을 제한하는 요소
  • 가장 느린 부분이 전체 속도를 결정
  • 개선이 필요한 핵심 지점
parsing
구문 분석:
  • 텍스트를 구조화된 형식으로 변환
  • 데이터에서 의미 추출
  • 기계가 이해할 수 있는 형태로 가공
noise
노이즈:
  • 필요 없는 부가 정보
  • 데이터 품질을 저하시킴
  • 분석을 방해하는 요소
lifeline
생명선:
  • 시스템이 작동하는 핵심 요소
  • 없으면 전체가 멈춤
  • 가장 중요한 기능
integrity
무결성:
  • 데이터가 변경되지 않고 유지
  • 일관성과 정확성 보장
  • 변조나 손상 방지
end-to-end
종단 간:
  • 시작부터 끝까지 전체 과정
  • 중간 단계를 건너뜀
  • 전체 흐름을 한 번에 처리
/goal
목표 설정 명령:
  • 어떤 것을 달성할지 정의
  • 방향성을 제시
  • 전체 작업의 목표점
/loop
반복 실행 명령:
  • 작업을 반복적으로 수행
  • 개선이 있을 때까지 계속
  • 품질을 높이기 위한 과정
harness
하네스:
  • 시스템을 제어하는 프레임워크
  • 작업을 안전하게 실행
  • 규칙과 프로토콜을 강제
protocol
프로토콜:
  • 통신 규칙과 표준
  • 데이터 교환 방식 정의
  • 상호 운용성 보장
routing
라우팅:
  • 작업을 적절한 대상에 할당
  • 경로를 결정하고 안내
  • 효율적인 분배
converge
수렴:
  • 목표 지점에 도달
  • 일관된 결과로 모임
  • 안정적인 상태에 이르름
cross-talk
상호 대화:
  • 에이전트 간 직접 소통
  • 서로 정보 공유
  • 협업을 통한 문제 해결
multi-agent
멀티 에이전트:
  • 여러 AI가 함께 일함
  • 각자 역할을 맡아 협업
  • 복잡한 작업을 분산 처리
trade-off
트레이드오프:
  • 하나를 얻으면 다른 것을 잃음
  • 장단점을 저울질
  • 최적의 균형점 찾기
skill
스킬:
  • 특정 작업을 수행하는 기능
  • 재사용 가능한 도구
  • AI가 활용하는 능력 단위
logical branch
논리적 분기점:
  • 중요한 의사결정 지점
  • 다른 경로로 나눠지는 곳
  • 기억해야 할 핵심 순간
4-layer memory
4계층 메모리 구조:
  • L1: 즉각적인 컨텍스트
  • L2: 최근 대화 기록
  • L3: 중요한 결정점
  • L4: 영구 지식 저장
schema
스키마:
  • 데이터 구조와 형식 정의
  • 일관된 형식 강제
  • 유효성 검사 기준
metadata
메타데이터:
  • 데이터에 대한 데이터
  • 생성 시간, 출처, 유형 등
  • 관리와 검색에 유용
fallback
폴백 메커니즘:
  • 실패시 대체 방안
  • 안전한 대안 제공
  • 시스템 중단 방지
algorithm
알고리즘:
  • 문제 해결을 위한 단계
  • 일관된 결과 보장
  • 자동화된 처리 절차
gate
게이트:
  • 통과 여부를 검사하는 지점
  • 품질 기준 강제
  • 잘못된 것 차단
mechanism
메커니즘:
  • 작동하는 방식과 원리
  • 시스템이 움직이는 구조
  • 자동화된 규칙 집합
