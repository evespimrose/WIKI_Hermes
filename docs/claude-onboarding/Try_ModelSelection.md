## 1. 현황 (As-Is)

- **주의력 분산** = 콘텍스트 윈도우 크기 vs 효율성 트레이드오프 불명확

## 2. 모델별 특성 분석

### 2-A. 아키텍처 명확성 (행동 일관성 & 예측 가능성)

| 모델 | 특성 |
|------|------|
| **Fable 5** | • 새로운 frontier 모델 · 최고 성능, 최고 비용 · adaptive thinking **기본값** (자동 활성화) · 시스템 프롬프트 순응도 **극도로 높음** |
| **Opus 4.8** | • 성숙한 설계 (4.7 기반) · 4.7과 API 호환 (breaking change 0) · 예측 가능한 콘텍스트 관리 · 프로덕션 기준선 |
| **Sonnet 4.6** | • 균형 모델 · 속도↔지능 트레이드오프 명확 · `effort` 파라미터 영향 큼 |
| **Haiku 4.5** | • 가장 단순 · 200K 제약 명확 · 기능 제약 명시적 |

### 2-B. 토큰 사용량 프로필 (콘텍스트 관리 능력)

| 모델 | 입력 토큰 | 행동 특성 |
|------|---------|---------|
| **Fable 5** | **매우 높음** (+15~25% vs Opus 4.8) | • thinking 생략 불가 (explicit `disabled` → 400 error) · 사고 과정 생략되지만 보다 깊은 추론 · 동일 작업 토큰 오버헤드 심각 |
| **Opus 4.8** | **기준선** | • adaptive thinking (권장) · 생각→액션 최적화 · 토큰 예산 수립 가능 |
| **Sonnet 4.6** | **effort에 따라 가변** | • `effort: low` → 최소화 · `effort: high` → Opus 근접 |
| **Haiku 4.5** | **최소** | • 200K 컨텍스트 제약 · 추론 제한적 |

### 2-C. 규칙 순응 & "탈옥빈도" (사용자 지시 회피 경향)

**"탈옥" = 사용자 Rule/Hook 무시 + 주의력 분산 (자의적 행동)**

| 모델 | 순응도 | 특징 |
|------|--------|------|
| **Fable 5** | ⚠️ **극도로 높음** (탈옥 확률 거의 0) | • CRITICAL/MUST 언어 → 과도하게 해석 · 사용자 의도 강하게 추론 · 무시할 Rule 거의 없음 · **주의력 분산 리스크**: 과도한 thinking으로 인한 "overthinking" (비용↑) |
| **Opus 4.8** | ✅ **균형** (탈옥 확률 ~5%) | • 명시적 지시 충실 · 모호한 지시는 질문으로 돌려줌 · 일부 소수 결정은 자율 판단 |
| **Sonnet 4.6** | ✅ **신뢰할 수 있음** (탈옥 확률 ~10%) | • 구조화된 지시 선호 · effort 낮음 = 더 직관적 · 과도한 검증 회피 가능 |
| **Haiku 4.5** | ⚠️ **제약 多** (탈옥 확률 ~20~30%) | • 복잡한 rule chain 이해 불가 · 장문 명령어 무시 경향 · 간단한 지시만 신뢰 |

**"주의력 분산" 사례:**

```
명령어: "다음 작업을 정확히 수행하고 다른 것은 하지 마시오"

Fable 5: 명령어 과해석 → 추가 검증, 예외 처리 추가 → 범위 확대
Opus 4.8: 명령어 문자 그대로 이행
Sonnet 4.6: 명령어 이행 + 자의적 개선 시도 (effort:high일 때)
Haiku 4.5: 명령어 중간 부분만 이행 (복잡도 때문)
```

### 2-D. 단기 비용 (운영 비용 예측 가능성)

| 측면 | Fable 5 | Opus 4.8 | Sonnet 4.6 | Haiku 4.5 |
|------|---------|----------|-----------|-----------|
| **입력 가격** | $10/MTok | $5 | $3 | $1 |
| **출력 가격** | $50/MTok | $25 | $15 | $5 |
| **동일 작업 토큰** | Opus +20% | 기준 | -30~50% | -60~80% |
| **동일 작업 비용** | Opus ×2.0 | 기준 | ×0.45~0.7 | ×0.15~0.3 |
| **예산 예측 가능** | 어려움 (overthinking) | 높음 | 높음 | 높음 |
| **가격 변동 리스크** | 높음 (frontier, 초기) | 낮음 (GA) | 낮음 (GA) | 낮음 (GA) |

**예시 계산:**

```
작업: 1000자 요약 (1M 컨텍스트, effort:high, thinking enabled)

Fable 5:
  input: 50k tokens × $10/MTok = $0.50
  output: 2k tokens × $50/MTok = $0.10
  total: $0.60

Opus 4.8:
  input: 42k tokens × $5/MTok = $0.21
  output: 2k tokens × $25/MTok = $0.05
  total: $0.26

Sonnet 4.6 (effort:high):
  input: 38k tokens × $3/MTok = $0.11
  output: 2k tokens × $15/MTok = $0.03
  total: $0.14

Haiku 4.5 (effort:low):
  input: 12k tokens × $1/MTok = $0.01
  output: 2k tokens × $5/MTok = $0.01
  total: $0.02
```

**결론**: Fable 5는 동일 작업 비용 **2배**, Sonnet 4.6 대비 **4배**

#### 🏆 **기본 추천: Opus 4.8**

**이유:**
- 성숙한 설계 (4.7 기반, API 호환)
- RX_1 기존 코드 마이그레이션 0 비용
- 규칙 순응도 높음 (탈옥 ~5%)
- 토큰 예측 가능 (thinking adaptive)
- 비용 vs 지능 균형

**RX_1 프로젝트 적용:**
```python

thinking={"type": "adaptive"}
output_config={"effort": "high"}  # 복잡한 셰이더 코드에 최적
max_tokens=128000  # 스트리밍 필수
```

## 7. 미결 사항 연계

- **Hook & Rules 탈옥빈도**: Opus 4.8 vs Sonnet 4.6 실제 테스트 필요
  - 세션 내 복잡한 rule chain 준수도 측정
  - 주의력 분산 (자의적 범위 확대) 빈도 기록

- **토큰 사용량 변수**: 실제 프로젝트 프롬프트로 count_tokens() 재측정
  - 각 모델별 thinking 생각 깊이 차이
  - effort 파라미터별 토큰 편차

- **콘텍스트 윈도우 충분성**: Haiku 4.5 (200K) 평가
  - RX_1 렌더링 파이프라인 전체 컨텍스트 크기 측정
