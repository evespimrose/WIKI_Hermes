# Claude 중급 가이드

## 1. 트랜스포머(Transformer) 개념

트랜스포머는 현대 LLM의 기반이 되는 신경망 구조입니다.

### 1.1 인코더(Encoder)
입력 데이터를 이해하고 특징을 추출하는 부분입니다.

### 1.2 디코더(Decoder)
추출된 특징을 바탕으로 출력을 생성하는 부분입니다.

### 1.3 인코더-디코더 구조
둘을 함께 사용하는 구조로, 번역 작업 등에 적합합니다. (예: T5)

## 3. 모델 분류

### 3.1 생성형(Generative) vs 판별형(Discriminative)
- **생성형**: 새로운 콘텐츠를 만듦 (예: GPT, Claude, DALL-E)
- **판별형**: 분류/판단만 함 (예: 이미지 분류 모델)

### 3.2 추론 모델 vs 비추론 모델
- **추론 모델**: 단계별로 생각하며 답변 (예: Claude with Chain-of-Thought)
- **비추론 모델**: 직접 답변 생성

### 3.3 대규모 모델 vs 소규모 모델
- **대규모**: 높은 성능, 높은 비용 (예: GPT-4, Claude Opus)
- **소규모**: 빠르고 저렴, 특정 작업에 최적화 (예: Llama 3 8B)

### 3.4 폐쇄형(Closed) vs 개방형(Open)
- **폐쇄형**: API로만 접근, 내부 공개 안 됨 (예: GPT, Claude)
- **개방형**: 가중치 공개, 자유롭게 수정 가능 (예: Llama, Mistral)

### 3.5 Auto-regressive LLM vs dLLM

**동작원리 차이:**
- **Auto-regressive**: 이전 토큰 기반으로 다음 토큰을 **순차적**으로 예측 (대부분의 LLM, ex: GPT, Claude)
- **dLLM (Diffusion LLM)**: 노이즈를 점진적으로 제거하는 **확산 프로세스**로 텍스트 생성 (초기 랜덤 노이즈 → 점진적으로 의미 있는 텍스트로 변환)

### 3.6 언어 모델 vs 동작 모델
- **언어 모델**: 텍스트 이해/생성 (예: GPT, Claude)
- **동작 모델**: 행동 계획/실행 (예: 로봇 제어 AI)

## 4. 토큰(Token)이란?

토큰은 Claude가 텍스트를 처리하는 기본 단위입니다. 대략 **4개의 영문자 = 1토큰**, **한글 1자 = 1~2토큰** 정도로 생각하시면 됩니다.

예:
- "Hello" → 1토큰
- "안녕하세요" → 2~3토큰

### 4.1 각 Claude 모델의 스펙과 특성 (Try_ModelSelection.md 기준)

#### 기본 스펙

| 모델 | 컨텍스트 | 최대 출력 | 입력 가격 | 출력 가격 |
|------|---------|---------|---------|---------|
| **Fable 5** | 1M | 128K | $10/MTok | $50/MTok |
| **Opus 4.8** | 1M | 128K | $5/MTok | $25/MTok |
| **Sonnet 4.6** | 1M | 64K | $3/MTok | $15/MTok |
| **Haiku 4.5** | 200K | 64K | $1/MTok | $5/MTok |

**토큰 변환 기준:**
- 영문: 약 4글자 = 1토큰
- 한글: 약 1.3~2글자 = 1토큰
- 1000자 요약 → 약 250~770토큰 (영문/한글 차이)

---

#### 단기 비용 (운영 비용 예측 가능성)

| 측면 | Fable 5 | Opus 4.8 | Sonnet 4.6 | Haiku 4.5 |
|------|---------|----------|-----------|-----------|
| **입력 가격** | $10/MTok | $5 | $3 | $1 |
| **출력 가격** | $50/MTok | $25 | $15 | $5 |
| **동일 작업 토큰** | Opus +20% | 기준 | -30~50% | -60~80% |
| **동일 작업 비용** | Opus ×2.0 | 기준 | ×0.45~0.7 | ×0.15~0.3 |

**결론**: Fable 5는 동일 작업 비용 **2배**, Sonnet 4.6 대비 **4배**

## 5. 컨텍스트(Context) & 세션(Session)

### 5.1 컨텍스트
현재 대화에서 Claude가 기억하고 있는 정보의 총합입니다. 대화가 길어질수록 컨텍스트도 커집니다.

### 5.2 세션
하나의 대화 단위를 의미합니다. 새로운 세션을 시작하면 이전 대화 내용은 초기화됩니다.

## 7. Claude 기본 Tool 설명

Claude가 파일과 시스템과 상호작용하는 기본 도구들입니다. (대부분 windows 기준 windows powershell 명령어로 동작)

| Tool | 설명 |
|------|------|
| **Read** | 파일 내용 읽기 |
| **Write** | 새 파일 생성 또는 기존 파일 덮어쓰기 |
| **Edit** | 기존 파일 내용 일부 수정 (정확한 문자열 매치 기반) |
| **Delete** | 파일 또는 폴더 삭제 |

| **LS** | 폴더 내용 목록 보기 |
| **Run** | 명령어 실행 (터미널 명령) |
| **Grep** | 텍스트 내용으로 파일 검색 |

## 8. Claude CLI vs 데스크탑 vs 웹

| 플랫폼 | 장점 | 단점 |
|--------|------|------|
| **웹** | 브라우저만 있으면 됨, 업로드 중심, 제약 많음 | 속도 제한, 대용량 파일 처리 불편 |

## 9. 서브에이전트(Sub-agent) 개념

서브에이전트는 메인 에이전트가 분할한 작업을 담당하는 보조 AI 에이전트입니다.

### 9.1 활용 예시
- **사용주체**: 메인 Claude가 복잡한 프로젝트를 분석할 때
  1. 메인 에이전트가 작업을 여러 하위 작업으로 분할
  2. 각 서브에이전트가 특정 부분 담당 (예: 코드 분석, 문서 작성, 테스트 실행)
  3. 결과를 메인 에이전트가 종합

## 10. 엔지니어링 개념

### 10.1 프롬프트 엔지니어링(Prompt Engineering)
AI에게 어떤 질문/명령을 내릴지 설계하는 것입니다. 명확하고 구체적인 프롬프트가 좋은 결과를 만듭니다.

### 10.2 컨텍스트 엔지니어링(Context Engineering)
AI에게 제공할 배경 정보를 선별하고 구조화하는 것입니다. 관련성 높은 정보만 제공하는 게 핵심입니다.

### 10.3 하네스 엔지니어링(Harness Engineering)
AI를 특정 워크플로우에 통합하고 자동화하는 시스템을 구축하는 것입니다.

### 10.4 AI 입출력: 마크다운 vs HTML
AI가 텍스트를 출력할 때 주로 사용되는 두 형식의 장단점을 비교합니다.

| 구분 | 마크다운(Markdown) | HTML |
|------|-------------------|------|
| **장점** | - 간결하고 가독성 좋음<br>- 토큰 사용량 적음<br>- 구조화 쉬움(제목, 목록, 표 등)<br>- 버전 관리 용이 | - 풍부한 스타일링(색상, 폰트, 애니메이션)<br>- 상호작용 기능(버튼, 폼, 링크)<br>- 정교한 레이아웃 제어<br>- 웹에서 직접 표시 가능 |
| **단점** | - 복잡한 스타일링 어려움<br>- 상호작용 기능 제한<br>- 정교한 레이아웃 불가 | - 토큰 사용량 많음(태그로 인해)<br>- 가독성 떨어짐(태그가 많아서)<br>- 복잡하고 작성 시간 오래 걸림 |

**사용 시기:**
- **마크다운**: 문서, 노트, 코드 설명, 간단한 내용 공유
- **HTML**: 웹 페이지, 인터랙티브 문서, 정교한 디자인 필요한 경우

## 11. 오케스트레이터(Orchestrator) 개념

### 11.1 인간 오케스트레이터
사용자가 직접 작업을 계획하고 AI를 지휘하는 방식입니다. 복잡한 의사결정이 필요할 때 유용합니다.

### 11.2 에이전트 오케스트레이터 (with 서브에이전트)
AI 자체가 작업을 분배하고 서브에이전트를 관리하는 방식입니다. 반복적이고 규칙적인 작업에 적합합니다.
