## 1. 현황 (As-Is)

### 1.1 파일 구조 — "실증 프로젝트"의 실제 정체

`claude-personal-integrated-workflow`는 **유토피아 3대 기둥 중 2개만 담은 메타-툴링 저장소**다. 실측 구조:

| 구성 | 실체 | 로드맵 기둥 |
|---|---|---|
| `src/ __tests__/ dist/ site/ scripts/` | **codegraph 엔진 포크** (TS, tree-sitter, MCP 서버) | 2.2 LLM 위키(코드그래프 벡터DB) |
| `.claude/ .codex/ .trae/ .agents/` | 워크플로 스캐폴딩 (agents 18 / hooks 16 / skills 12 / commands 11) | 2.3 AI(서브에이전트·메모리) |
| `docs/ memory/ manage/` | 설계철학·배포·검증·try·output 문서 | 2.2 / 2.3 |
| **(없음)** | Unity·asmdef·DOTS·Burst·VContainer 코드베이스 | **2.1 로컬 코드베이스 ❌** |

로드맵 사본은 `docs/output/Workflow_End_To_End_LoadMap.md`에 존재하며 WIKI 원본과 **byte-identical**(diff 0). 즉 실증 프로젝트는 로드맵을 내장하고 있으나 그 2.1 기둥을 구현하지는 않는다.

### 1.3 결정적 발견

1. **"실증 프로젝트"는 게임이 아니라 워크플로 배포본이다.** 로드맵 2.1·3.1(Unity+DOTS+Burst+VContainer, asmdef Clean Arch, BLK dictionary grep)의 **물리적 실체가 이 저장소엔 0%**. 이 기둥의 실증은 WIKI 볼트의 `RX_1/*` 노트(개념)와 별도 Unity 프로젝트(저장소 외부)에만 존재.
2. **구현된 에이전트 토폴로지 ≠ 로드맵 §6 토폴로지.** 실제는 RX_1 게임 스튜디오(producer·creative/technical-director·lead·unity-specialist·도메인전문가). 로드맵 §6은 일반화된 *Mastermind→(Prototype↔Adversarial)→QualitySentinel→Reporter* 4-스테이지. 교집합은 quality-sentinel·reporter·prototyper뿐. **Mastermind·Adversarial 2개 핵심 역할이 에이전트로 미존재.**
3. **로드맵이 스스로 지목한 생명선(§4·§8 = Conclusion 파싱 무결성·강제 훅)이 가장 덜 구현됨.** 스키마·메타데이터·fallback 전무, PreCompact 훅 미등록.
4. **로드맵 §5 궁극 UX("~~해줘"→자율팀 end-to-end)는 진입점(`/goal`·team flag)부터 부재.** `/loop`만 존재.

## 3. 장단점 비교

### 3-A. 아키텍처 명확성
- (+) 5개 결손을 "볼트→스캐폴딩→훅" 순으로 정렬하면 SSOT 원칙(L5가 정책 단일출처)과 일치.
- (−) Mastermind 팀과 RX_1 스튜디오가 **이중 토폴로지**로 공존 → 어느 쪽이 정본인지 정의하지 않으면 환각 원천(로드맵 6대원칙 "단일책임" 위반).

### 3-B. 컴파일·빌드 영향
- codegraph 엔진(src/) 변경 시 `npm run build`로 schema.sql·wasm 복사 필수(CLAUDE.md house rule). Conclusion 노드등록을 codegraph MCP 툴 경유로 하면 엔진 코드 변경 없이 가능 → 빌드 영향 0.
- 2.1 Unity 번들을 실제로 넣으면 저장소 성격이 폭증(수백 MB, .meta, asmdef) → 빌드/인덱싱 비용 급증. **스코프아웃이 빌드 관점 우월.**

### 3-C. 현재 작업과의 연관성
- 메모리 노트 `codegraph-first-search-policy`와 충돌 검토 필요: 로드맵·철학문서 §8은 여전히 *dictionary-first/first-grep* 중심. 볼트 정본은 *codegraph-first + dictionary-fallback*. **로드맵 텍스트가 구식 정책을 반영** → 파리티 추구 시 어느 방향에 맞출지 결정 필요(§6 미결).

### 3-D. 단기 비용
| 위험 | 항목 | 근거 |
|---|---|---|
| **고위험** | 2.1 Unity 기둥을 "100% 동일"하게 충족하려 실 게임코드 번들 | 저장소 정체성 변질·인덱스 폭발·유지 불가. 로드맵의 "100% 동일"이 문자적으로 불가능한 지점 |
| **고위험** | Mastermind 완전자율(§5 "탈옥없이 end-to-end") | 무한 런타임 권한 = Manifesto 스스로 "자살행위"로 규정. 인간 Confirm 게이트 유지 필수 |
| **중위험** | EXPERIMENTAL_AGENT_TEAM 크로스토크 | §7 멀티에이전트 대화 +30% 토큰 트레이드오프(볼트 기록 존재). "초미세토큰" 목표와 정면충돌 |
| **중위험** | Conclusion 스키마 강제 | 너무 엄격하면 파싱실패 폭증→fallback 남발→인덱스 오염(로드맵 §4가 경고한 바로 그 실패모드) |
| **저위험** | 볼트 5개념 원자노트 작성 | 문서 작업, 코드 영향 0 |
| **저위험** | PreCompact 훅 등록 | 스크립트 이미 존재, settings 한 줄 |

### 3-E. 장기 유지보수
- (+) 볼트 선기록은 정책-구현 드리프트를 구조적으로 차단 → 장기 무결성↑.
- (−) 이중 토폴로지·구식 dictionary-first 잔재를 방치하면 매 세션 주입(L1)마다 모순 컨텍스트가 환각을 누적.

## 4. 순환 의존성·방향 위험 분석

1. **정책↔구현 순환**: 로드맵(정책)이 *dictionary-first*를 말하나 볼트 정본은 *codegraph-first*. 로드맵을 정본으로 삼아 구현을 맞추면 볼트 정본과 역행. **방향 강제**: 볼트(L5)가 최상위 SSOT이므로 로드맵 텍스트를 볼트에 맞춰 갱신해야지 그 반대가 아니다. → 파리티 작업 전 **로드맵 자체를 codegraph-first로 정정**하는 선결 과제 발생.
2. **Conclusion→codegraph→Conclusion 순환 위험**: 결론을 노드 등록 → 다음 세션이 그 노드를 읽어 또 결론 생성. 키워드 정규화 실패 시 자기참조 오염 루프. fallback `flag=unparsed`로 단방향 차단 필요.
3. **Mastermind→하위에이전트→Mastermind 보고 순환**: 크로스토크(team flag)가 컨텍스트 격리를 깨면 토큰 폭발. §7.1 "주방 작업대 원칙"(BLK/API만 교환)으로 단방향 프로토콜 강제 필요 — 이 원칙은 볼트에 존재(`서브에이전트_출력압축_원리`, `하네스_엔지니어링`)하나 **에이전트 통신 레이어에 미적용**.

## 5. 종합 평가 및 추천

### 결론 (한 문단)

실증 프로젝트는 로드맵 **2.2(코드그래프 위키)·2.3(AI 워크플로) 기둥을 견고히 구현**했으나, **2.1(Unity 코드베이스) 기둥은 0%**, 로드맵이 스스로 시스템의 *생명선*으로 지목한 **§4·§8(Conclusion 파싱 무결성·강제 훅)은 가장 미성숙**, **§5-6 Mastermind 자율팀은 진입점부터 부재**다. "100% 동일"은 문자적으로 달성 불가(2.1은 별 저장소, §5 완전자율은 Manifesto가 금기로 규정)이므로, 목표를 **"로드맵의 *구현 가능한 주장* 100% 충족 + 비현실 항목은 명시적 스코프아웃·정책정정"**으로 재정의하고, 위키 정본(L5)을 먼저 채운 뒤 스캐폴딩·훅을 충전하는 6페이즈를 권한다. 가장 큰 가치는 §8 Conclusion 표준화에 있으며 이것이 로드맵의 진짜 병목이다.

### 단계별 추천 순서

- **Phase 0 — 정합·스코프 확정 (코드 0)**: ① 2.1 기둥을 "레퍼런스 Unity 번들" vs "스코프아웃+RX_1 정본 링크" 중 택일(권장: 후자). ② 로드맵의 *dictionary-first* 표현을 볼트 정본 *codegraph-first*에 맞춰 정정(순환#1 해소). ③ 이중 토폴로지(Mastermind vs RX_1 스튜디오) 중 정본 결정.
- **Phase 1 — 볼트 결손 5개념 원자노트 (L5 SSOT 우선, 저위험)**: 아래 "로드맵 有·볼트 無" 5개를 원자노트로 기록 + 양방향 링크. 구현 전 정책 확정.
- **Phase 2 — Conclusion 표준화 (§8.1, 최고가치)**: `schema/conclusion.schema.json`(메타+키워드), output-arm 결론 라이터, **codegraph 노드 자동등록**, 파싱실패 fallback. 로드맵 병목 직접 해소.
- **Phase 3 — 압축 무결성 훅 (§8.2-8.3)**: settings에 `PreCompact` 등록(스크립트 기존재), 0-유실 키워드 자동추출, 압축 전후 검증 게이트.
- **Phase 4 — Mastermind 팀 (§5-6)**: `mastermind-orchestrator`·`adversarial-verifier` 에이전트, `/goal`(또는 /loop 래퍼), `EXPERIMENTAL_AGENT_TEAM` flag — 단, §7 +30% 토큰·§4 인간 Confirm 게이트를 동시 강제(주방 작업대 프로토콜 통신레이어 적용).
- **Phase 5 — 2.1 기둥 처리**: Phase 0 결정에 따라 레퍼런스 Unity 번들 추가 또는 RX_1 정본을 실증 문서에 명시 링크 + `manage/dictionary.md` 인스턴스(자기 저장소용 또는 포인터).
- **Phase 6 — 비현실 지표 검증 하네스**: 500-600턴 0-유실·토큰 90/95/98% 절감·"~~해줘" end-to-end UX를 측정 스크립트로 계측(현재 전부 미계측 주장).
