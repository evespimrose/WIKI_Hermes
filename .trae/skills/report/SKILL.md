# Report — Git 저장소 심층 분석 + 입국심사

## 역할

git 주소(또는 로컬 저장소 경로) 하나를 받아 **정확히 두 개의 마크다운 문서**를 생성한다:

1. **심층 분석 보고서** → `docs/analysis/<slug>_analysis.md` — [[try]] 스킬의 엄정성으로 작성, 5개 항목 전부 포함.
2. **입국심사 문서** → `raw/<slug>_입국심사.md` — `compile-wiki a`(정합성 평가) 모드의 기준으로 작성.

작업이 끝나면 **"작업 완료" 네 글자만 출력**한다. 그 외 어떤 요약·보고·설명도 쓰지 않는다(절대조건, 맨 아래 참조).

이 두 문서의 분업: 분석 보고서는 *깊이 파는 의사결정 자료*(docs/에 영속), 입국심사는 *볼트 편입 후보 + 게이트 판정*(raw/에 적재되어 추후 compile-wiki 대상).

## Vault 감지

이 스킬은 LLM Wiki 볼트 안에서 동작한다. 시작 전에 작업 디렉토리(CWD)가 볼트 루트인지 검증한다:

1. CWD에 `raw/`와 `obsidian/`이 동시에 존재하면 볼트 루트.
2. 없으면 상위로 최대 3단계 탐색. 그래도 없으면 `~/.claude/llm-wiki-config.json`의 `vault_root` 사용.
3. 끝내 못 찾으면 사용자에게 볼트 경로를 확인한다(임의 생성 금지).

`docs/analysis/`가 없으면 생성한다. `raw/`는 항상 존재한다(없으면 볼트가 아님).

## 볼트 방향성 사전 로드 (4번·입국심사의 근거)

분석 4번 항목과 입국심사는 "이 볼트가 무엇을 지향하는가"를 알아야 판단할 수 있다. 시작할 때 다음을 읽어 기준을 잡는다:

- `schema/vault_mandate.md` — 볼트 지향 북극성(in-scope / out-of-scope). 있으면 *1차 기준*.
- 없으면 `obsidian/`의 허브 노트(`LLM_워크플로우_생태계`, `*_허브` 행성들)와 태그 분포에서 지향을 추론.

이 기준 없이 4번·입국심사를 쓰면 근거 없는 점수가 된다 — 반드시 선행한다.

## 플래그: `-a` (Atomic Mode)

이 스킬은 선택적으로 `-a` 플래그를 받는다:
- **없음 (기본)**: 분석 보고서 + 입국심사 1순서 (✅승인 / 🟡조건부 / ⛔보류).
- **`-a` 포함**: 같은 분석 + 원자 단위 분해 + **사용자 체크리스트 입국심사** (각 원자별 선택 박스, 5단계 추천도, 토큰 경제학 분석).

### 사용 예시

```
기본:  /report https://github.com/user/project
원자:  /report -a https://github.com/user/project
```

### 워크플로우

**기본 모드**:
```
분석 (5섹션) → 입국심사 (5기준 판정) → 최종 verdict (승인/조건부/보류)
```

**원자 모드**:
```
분석 (5섹션) → 원자 분해 (3~8개) → 입국심사 (각 원자별 체크리스트 + 추천도 + 토큰 분석)
                                ↓
                    사용자가 [x] / [ ] 선택
                                ↓
                    선택 결과 바탕으로 /compile-wiki 호출 또는 최종 판정 입력
```

원자 모드는 "내 워크플로우에 맞는 것만 채택하고 싶다"는 사용자를 위해 **세밀한 제어**를 제공한다.

## Phase 1 — 저장소 확보 + 플래그 감지

입력이 git URL인지 로컬 경로인지, `-a` 플래그가 있는지 판별한다.

- **로컬 경로이고 존재**: 그대로 읽는다(클론 불필요).
- **git URL**: 임시 디렉토리에 얕은 클론.
  ```
  git clone --depth 1 <url> <temp>/<slug>
  ```
  `<slug>`는 저장소 이름(예: `github.com/org/harness` → `harness`).
- **클론 실패**(비공개·인증 필요·404): `gh repo view`를 시도하고, 그래도 막히면 WebFetch로 README와 디렉토리를 최대한 읽는다. 이 경우 *조사 깊이가 제한됨*을 두 문서 모두에 명시한다(특히 src/ 미열람 시 2·3번 신뢰도 하락을 솔직히 기록).

분석이 끝나면 임시 클론은 정리해도 좋다(원본 raw/obsidian은 절대 건드리지 않는다).

## Phase 1.5 — 원자 분해 (if `-a` 플래그)

**기본만 수행하고 싶으면 이 단계는 스킵.**

`-a` 플래그가 있으면, Phase 2 정독 후 README + src/ 코드를 스캔해 **기능별 원자 단위로 분해**한다.

### 원자 단위 식별 원칙

- **단위**: 하나의 독립적 기능·메커니즘·패턴. "이것만 떼어내 다른 워크플로우에 적용 가능한가?"가 기준.
- **범위**: 3~8개 원자 추출(너무 적으면 세분화 불필요, 너무 많으면 선택 피로).
- **명명**: 동작 동사형 + 명사(예: "캐시 레이어 자동화", "의존성 주입 메커니즘", "로그 압축 정책").

### 분해 예시

```
프로젝트: 헤르메스 스카우터
추출 원자:
1. 키워드 필터링(사전필터) — scout_lexicon 기반 pre-filter
2. 배치 입국심사(자동화) — raw/_scout → compile 자동화
3. 환류 루프(학습) — 판정 → scout_lexicon 가중치 갱신
4. RAG 스토리지(외부화) — MD 메모리 내보내기
```

각 원자는 이름만 임시 기록(상세는 Phase 4 체크리스트에서).

## Phase 2 — 필수 정독

추상화·검증은 전체 맥락을 잡은 뒤에만 신뢰할 수 있다. 다음은 *건너뛰기 금지*:

- **README.md (필수)** — 프로젝트가 *스스로 표방하는 목적*. 1번의 1차 근거.
- **src/ 또는 핵심 코드베이스 (필수)** — *실제 구현*. 2·3번의 1차 근거. 전부 읽을 필요는 없으나 진입점·핵심 모듈·설정(package.json/pyproject 등)은 반드시 본다. 토큰 경제를 위해 디렉토리 구조부터 보고 핵심 파일을 표적 정독한다.
- 보조: CHANGELOG, docs/, 테스트(주장 검증에 유용), 이슈/TODO/스텁(과장 탐지에 유용).

README가 없으면 그 사실 자체가 1번의 발견이다(자기서술 부재). src/가 없으면 실제 구현 위치(루트 .ts/.py 등)를 찾아 대체한다.

## Phase 3 — 문서 1: 심층 분석 보고서

`docs/analysis/<slug>_analysis.md`에 저장. [[try]]의 정신(현 상태 정직 조사 → 적대적 검증 → 정량 결론)을 따르되, 아래 5개 섹션을 *정확히 이 순서로 전부* 채운다. 하나라도 비우지 않는다.

```markdown
# Report — <프로젝트명> 분석

> 생성일: <오늘 YYYY-MM-DD>
> 대상: <git URL 또는 경로> (<커밋 해시 또는 clone 시점>)
> 조사 범위: README ✅ / src ✅ (또는 제한 사유)

## 1. 무엇을 위해 개발되었는가 (목적)
- README가 표방하는 목적·문제정의·타깃 사용자.
- 한 줄 정의 + 근거(README 인용은 짧게).

## 2. 어떻게 달성했는가 (구현)
- 핵심 아키텍처·메커니즘을 src/ 실제 코드 근거로 서술.
- 진입점 → 핵심 모듈 → 데이터/제어 흐름. 파일 경로로 짚는다.

## 3. "정말로" 그러한가 (적대적 검증)
- 1·2번의 주장을 *의심하며* 코드와 대조한다. README의 약속과 실제 구현의 간극을 찾는다.
- 점검: 핵심 기능이 실제로 구현됐는가(스텁·TODO·미완 아님?) / 테스트가 주장을 뒷받침하는가 / 과장·마케팅 표현이 코드로 입증되는가 / 숨은 전제·한계.
- 정직하게 쓴다. "표방대로 작동" 또는 "X는 표방하나 실제는 Y" 형태로 결론.

## 4. LLM 위키 워크플로우에 무엇을 기여할 수 있는가
- `schema/vault_mandate.md`의 지향과 대조해, 이 프로젝트가 본 볼트의 워크플로우(토큰통제·에이전트 오케스트레이션·메모리·위키 메타시스템 등)에 줄 수 있는 구체적 기여.
- 관련 기존 노트와 연결([[...]]). 기여 없으면 "없음"과 사유.

## 5. 핵심 로직 100% 적용 시 턴당 토큰 증가 추정
- 이 프로젝트의 핵심 로직을 본 워크플로우에 *상시(always-on)* 적용했을 때 턴당 고정 토큰이 기존 대비 몇 % 증가하는지 정량 추정.
- 방법: ①상시 주입/소비 요소 식별(hook·메모리·멀티에이전트 오버헤드·인덱스 첨부 등) ②기존 턴당 베이스라인 가정 ③증가분 산출. 가정과 근거를 명시한다.
- 런타임 상시 footprint가 없는 일회성 도구면 그 사실을 밝히고 *호출당* 비용으로 대체 추정.
- 결론: `+N% (가정: …)` 형태. 본 볼트가 토큰 강박(−62% 최적화 역사)임을 고려해 트레이드오프를 한 줄 평한다.

## 종합
- 한 단락: 채택 가치 + 핵심 조건/리스크.
```

## Phase 4 — 문서 2: 입국심사

`raw/<slug>_입국심사.md`에 저장. raw/에 들어가므로 **raw 프론트매터**를 갖춰 파이프라인 추적이 가능하게 한다.

### 기본 모드 (플래그 없음)

`compile-wiki a`(정합성 평가) 모드의 5대 기준과 판정을 그대로 적용:

```markdown
---
source_type: git
source_url: <git URL>
captured: <오늘 YYYY-MM-DD>
status: raw
verdict: approve | conditional | hold
analysis_mode: standard
---

# 입국심사 — <프로젝트명>

## 지향 방향성(기준)
- `schema/vault_mandate.md`(또는 추론) 요약 한 줄.

## 핵심 개념 추출(가상 원자화)
- 이 저장소가 컴파일되면 생길 법한 원자 노트 후보 3~6개(저장은 안 함).

## 5대 기준 판정
- 방향성 적합도(Fit) N/5 — …
- 신규성(Novelty) N/5 — …
- 연결성(Linkage) N/5 — 링크 가능한 기존 노트 …
- 충돌(Conflict) — 있으면 '논쟁 및 관점 차이' 후보 표시
- 품질·출처(Quality) N/5 — provenance 명확성

## 판정
- ✅승인 / 🟡조건부(+조건) / ⛔보류(+사유·대안)
- 예상 생성/확장 노트 + 링크 후보.

## 분석 보고서 연계
- 상세는 `docs/analysis/<slug>_analysis.md` 참조.
```

판정은 분석 보고서(특히 3·4번)와 정합해야 한다 — 적대적 검증에서 과장이 드러났다면 Quality·Fit에 반영한다.

### 원자 모드 (with `-a` 플래그)

`-a` 플래그가 있으면 기존 5대 판정 대신 **원자 단위 체크리스트**를 생성. 각 원자마다:
- **선택 박스**: 사용자가 `[ ]` / `[x]` 로 토글 가능
- **5단계 추천도**: 1~5 별 + 이유
- **토큰 경제학**: 이 원자 100% 적용 시 turn당 토큰 증가 + risk/benefit + 조건부 추천

```markdown
---
source_type: git
source_url: <git URL>
captured: <오늘 YYYY-MM-DD>
status: raw
verdict: pending_selection
analysis_mode: atomic
atoms_total: N
---

# 입국심사 — <프로젝트명> (원자 선택 모드)

> **사용자 가이드**: 아래 각 원자마다 `[ ]`/`[x]`로 선택. 
> 당신의 워크플로우를 위해 무엇을 채용하고 무엇을 제외할지 선택하세요.
> 토큰 비용 · 리스크 · 기여도를 고려해 의사결정을 지원합니다.

## 지향 방향성(기준)
- `schema/vault_mandate.md` 요약 및 추론 근거.

## 원자 단위 체크리스트 (선택 가능)

**추천도**: ⭐⭐⭐⭐⭐ 5/5 (또는 1/5)  
**기여도**: 
- 워크플로우 적합도: 매우 높음 / 높음 / 중간 / 낮음 / 없음
- 신규성: 이미 있음 / 약간 다름 / 새로운 패턴 등

**토큰 경제학**:
- **턴당 비용**: +3% (기존 베이스 가정: 호출당 22K token)
- **조건**: 상시(always-on) 적용 / 온디맨드(호출당) / 일회성(setup only)
- **Benefit**: 높음 (토큰 대비 효율)
- **Risk**: 낮음 (호환성 / 복잡도 / 유지보수)

**근거**:
- 기여 이유: [구체적 설명]
- 토큰 비용 계산: [가정 + 산식]
- 리스크 평가: [위험 요소]

**판정 코멘트**:
[이 원자가 좋은 이유 / 주의할 점 / 조건 등 1~2문단]

---

[동일 구조 반복]

---

## 종합 평가

| 지표 | 평가 |
|---|---|
| 총 원자 수 | N개 |
| 추천도 5/5 원자 | N개 (총 +?% 토큰 cost) |
| 조건부(3/5) 원자 | N개 |
| 비추천(1~2/5) 원자 | N개 |
| **예상 채용 시나리오** | 전수용 (high integration) / 선별용 (pick & choose) / 교육용 (concepts only) |

## 최종 판정
- **정식 verdict**: 사용자 선택 후 재입력 대기 (pending_selection)
- **AI 추천**: [모든 원자 채용 / 상위 3개만 / 이 원자들만 제외 등 1문단 조언]

## 분석 보고서 연계
- 상세는 `docs/analysis/<slug>_analysis.md` 참조.
```

**중요**: 원자 체크리스트는 **사용자 선택 가능성**을 최우선. 모든 추천은 "조언"일 뿐 결정권은 사용자.

사용자가 선택을 완료하면 선택 결과를 바탕으로 최종 `verdict: approve | conditional | hold`를 업데이트하고 raw/에 재저장할 수 있다.

## 출력 절대조건

두 문서를 모두 저장한 뒤, 사용자에게는 **"작업 완료"** 네 글자만 출력한다.

- 무엇을 만들었는지, 어디에 저장했는지, 발견이 무엇인지 — *일절 쓰지 않는다*. 보고는 두 문서 안에 있고, 사용자가 직접 연다.
- 진행 중간 단계의 설명도 최소화한다. 도구 호출은 하되 서사적 중계는 하지 않는다.
- 이유: 이 스킬의 산출물은 *파일*이지 대화가 아니다. 화면 보고는 중복이며 사용자가 명시적으로 배제했다.

### 원자 모드에서의 출력 차이

**기본 모드**: 분석 보고서 + 입국심사(단일 verdict) → 파일 2개  
**원자 모드 (`-a`)**: 분석 보고서 + 입국심사(원자 체크리스트) → 파일 2개  

두 경우 모두 "작업 완료"만 출력. 원자 모드라고 해서 추가 보고는 하지 않는다. 체크리스트는 파일 안에만 있고, 사용자가 직접 선택한 후 필요하면 재입력(`/compile-wiki`)으로 최종 판정을 내린다.

## 절대 하지 말 것

- 원본 README·코드를 그대로 복사-붙여넣기(추상화 의무).
- 분석이 덜 끝났는데 두 문서를 저장하고 "작업 완료" 선언:
  - **기본 모드**: 5섹션·5기준 전부 채운 뒤에만.
  - **원자 모드**: 모든 원자의 추천도·토큰비용·근거 작성 완료 뒤에만.
- `obsidian/`·`archive/`·`schema/`·기존 `raw/` 파일 수정(이 스킬은 `docs/analysis/`와 신규 `raw/<slug>_입국심사.md`만 쓴다).
- 클론 실패를 숨기기(조사 제한은 두 문서에 명시).
- "작업 완료" 외의 마무리 멘트(절대조건 위반).
- 날짜 추측(시스템 오늘 날짜 사용).
- 원자 모드에서 원자를 너무 많이 추출(8개 초과 금지 — 선택 피로 → 의사결정 마비).
- 원자 모드에서 토큰 비용 분석을 생략(모든 원자가 비용 평가를 필요로 함 — "비용 없음"도 명시).
- 원자 설명을 너무 짧게(각 원자마다 "기여도" + "토큰 비용 계산식" + "리스크" 최소 3줄이상).
