# 구체 지시 우선 원칙

좁은 범위의 구체적 지시가 넓은 범위의 일반 규칙을 이긴다는 원리. 따라서 전역 규칙은 한 곳에만 적어선 안 되고, 그와 충돌할 수 있는 *하위 스킬·커맨드 본문에도 반복 주입*되어야 효력이 생긴다. 프롬프트/스킬 설계 전반에 적용되는 메타 원리.

## 자가인가(自家認可) 탈옥

모델은 충돌하는 두 지시를 만나면 *더 구체적이고 작업에 직접 연결된* 쪽을 따른다. [[Try_Skill_타당성_검증]] 스킬에서 이 현상이 드러났다:

```
전역 규칙 (CLAUDE.md):  "소스 접근 전 codegraph 우선" — 일반·광범위
스킬 지시 (try SKILL.md): "1. READ 코드베이스 파악" — 구체·직접
→ 모델은 스킬 지시를 따라 codegraph를 건너뛴다
```

스킬이 스스로 "이 작업에선 읽어도 된다"고 인가해버리는 것. "분석이니까 자유롭게 읽어도 된다"는 합리화가 여기 얹힌다. 전역 규칙이 *있었음에도* 하위 지시가 이겼다.

## 왜 구체가 일반을 이기는가

구체적 지시는 *현재 작업의 맥락*에 직접 묶여 있어 모델의 주의(attention)를 더 강하게 끈다. 일반 규칙은 세션 전역에 떠 있어 [[컴팩션_생존_전략]]이 다루는 attention drop에 더 취약하다. 좁은 지시일수록 "지금 이 일에 대한 말"로 읽혀 우선순위가 높아진다.

이는 [[LLM_통제_철학_6대원칙]] 원칙 3(Hooks > Prompts)을 *프롬프트 내부의 위계*로 확장한 것이다. 원칙 3이 "텍스트보다 훅"이라면, 본 원칙은 "텍스트끼리도 구체적인 쪽이 강하다"를 말한다.

## 해결: 반복 주입

전역 규칙을 세웠다고 끝이 아니다. 그 규칙과 충돌할 여지가 있는 모든 하위 산출물(스킬, 커맨드, 에이전트 정의)의 본문에 같은 규칙을 *다시* 박아야 한다.

```
CLAUDE.md          → codegraph 우선 (전역)
skills/try/SKILL.md → codegraph 우선 (재주입) ← 이게 없으면 덮인다
commands/*.md      → codegraph 우선 (재주입)
```

[[Workflow_Ecosystem_버전계보]] v3.5가 RIPER 7개 커맨드 전부에 `## Codebase Navigation` 섹션을 넣은 것이 바로 이 반복 주입의 대규모 적용 사례다.

## 이중 방어에서의 위치

본 원칙은 [[이중방어_Defense_in_Depth]]의 *soft 층*을 왜 손봐야 하는지 설명한다. hook(hard 층)만 고치고 스킬 텍스트(soft 층)를 방치하면, 모델은 여전히 스킬 지시를 따르려다 hook에 부딪혀 매 턴 마찰을 빚는다. 근본 원인(soft 공백)은 텍스트 재주입으로만 닫힌다.

## 일반 원리

> 전역 정책은 하위 스킬·커맨드 본문에 반복 주입해야 실효성이 생긴다. 한 곳에만 적으면 더 구체적인 하위 지시에 덮인다. 규칙의 강제력은 *작성 위치의 구체성*에 비례한다.
