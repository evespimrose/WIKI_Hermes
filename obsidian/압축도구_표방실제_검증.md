# 압축 도구 표방-실제 검증

압축 도구가 "무손실"을 *자인*해도 정확도는 *증명*하지 않는다. 도입 전 표방(마케팅)과 실제(코드·eval 문서)를 분리해 대조하는 적대 검증 패턴. caveman은 엔지니어링이 신중함에도(경계 보존·Auto-Clarity·정직한 eval 설계) 마케팅이 미측정 정확도와 cherry % 위에 서 있는 전형 사례라 교본으로 쓰기 좋다.

## caveman 표방-실제 4괴리

1. **"기술 정확도 100%"** ↔ eval은 fidelity를 *못* 잰다. `k`만 답해도 −99%로 '승리' = 정보 파괴를 보상. 정확도 보존의 실증은 0이다.
2. **"75% / up to 87% 절감"** ↔ 실측 평균 **65%**(범위 22–87%, 프롬프트당 n=1). 75/87은 단일 cherry.
3. **"+26pt accuracy" 논문 인용** ↔ 정작 자기 정확도를 한 번도 측정 안 한 도구의 근거로 사용(appeal to authority). 인용 ID는 `26-04`(4월)인데 본문은 "March" — 내적 모순도 있다.
4. **"cost go down forever"** ↔ 스킬 프롬프트·SessionStart 주입·매턴 앵커가 *입력*에 상주. 짧은 답엔 입력 오버헤드가 출력 절감을 초과할 수 있음(레포 자인: "output savings are not the full economic picture").

## 도입 전 체크리스트

- 정확도를 **측정**했는가, 아니면 *자인*만 했는가?
- % 절감은 **무엇 대비**인가([[압축효과_측정_정직성]] — terse 대조군 없이는 무의미)?
- 근거 자료가 **그 주장**을 입증하는가(정확도를 안 잰 도구가 정확도 논문을 인용하지 않는가)?
- **입력 비용**을 차감한 순효과인가?

## 볼트에서의 위치

model-self-compliance를 신뢰하지 않는 [[LLM_통제_철학_6대원칙]]과 "correctness > brevity"의 [[압축_안전경계]]를, *마케팅 주장* 검증으로 확장한다. [[위키_저장_5대필터]] 승격 심사가 "무엇을 남길까"라면 본 패턴은 외부 도구를 들일 때 "표방을 곧이곧대로 믿지 마라"다. 측정 정직([[압축효과_측정_정직성]])과 한 쌍 — 전자는 *내 주장*을, 본 노트는 *남의 주장*을 감사한다.
