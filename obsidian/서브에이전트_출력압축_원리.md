# 서브에이전트 출력 압축 원리

서브에이전트의 tool-result는 메인 컨텍스트에 **그대로(verbatim) 주입**된다. 따라서 산문 2k 토큰을 반환하는 서브에이전트는 위임할 때마다 메인 예산에서 2k를 깎는다. 한 세션에서 위임 20회가 누적되면 이것이 *컨텍스트 고갈과 완수의 갈림길*이 된다. 핵심 처방: **서브에이전트 출력 자체를 압축(~1/3)하고, 산문이 아닌 grep 가능한 출력 계약으로 받는다.** [[하네스_엔지니어링]]의 멀티에이전트 우위를 토큰 경제학으로 정량화한 원리.

## 왜 출력측인가

메인이 직접 탐색하면 탐색 비용도 메인 예산이다. 그래서 [[3Tier_에이전트_스튜디오]]는 탐색·편집·검증을 서브에이전트로 위임한다. 그러나 위임의 숨은 비용은 **반환값 재주입**이다 — 서브가 아무리 격리돼 일해도 결론은 메인 컨텍스트로 돌아온다([[위키_기반_에이전트_맥락공유]]·[[AI_네이티브_멀티에이전트_운영]]). 이 재주입분을 압축하지 않으면 위임의 토큰 이득이 상쇄된다. [[맥락_오염_방지]]가 *무엇을* 들일지의 게이트라면, 본 원리는 서브에이전트 결과를 *얼마나 촘촘히* 들일지의 게이트다.

## 출력 계약 (grep 가능 형식)

압축은 "짧게 쓰기"가 아니라 **기계 파싱 가능한 고정 스키마**로 받는 것이다. 역할별 계약 예:

- **탐색(investigator)**: `path:line — \`symbol\` — 짧은 노트` / 매칭 없으면 `No match.` — `path:\d+` 로 grep 가능, 파일경로-우선·라인번호 부착·심볼 백틱.
- **편집(builder)**: `path:range — 변경요약(≤10단어)` + `verified: <재독 OK | mismatch @ path:line>` / 또는 종결 토큰 `too-big.`·`needs-confirm.`·`ambiguous.`(첫 토큰이 실패 신호).
- **검증(reviewer)**: `path:line: <severity emoji> <문제>. <수정>.` + `totals: N🔴 N🟡 N🔵` / 이슈 없으면 `No issues.`

원칙: **출력에 fallback 신호를 내장**한다(종결 토큰). 형식이 곧 파싱 실패율 0의 보증이며, 산문 자유도를 의도적으로 희생한다.

## 볼트 기여 — Priority-1 갭 충전

로드맵 §8.1(Subagent Conclusion 파싱 표준화)·§2.3(Conclusion → 극소 마크다운 저장)·§7.1(주방 작업대 원칙: 에이전트 간 파일 통째 전달 금지, 좌표+API 명세만 교환)이 요구하나 노트가 없던 갭을 메운다. [[Producer_QualitySentinel_Reporter_게이트]]가 `git diff HEAD`만 보고 *전체 파일 읽기 금지*로 토큰을 아끼는 것과 같은 계열 — 게이트 간 전달을 최소 표면적으로 묶는다.

## 한계

압축 계약을 실제로 강제하려면 서브에이전트 정의 파일에 형식을 못박아야 한다(프롬프트 의존이면 drift 발생). 과도한 형식 강제는 표현력을 깎으므로 종결 토큰·fallback을 반드시 포함한다.
