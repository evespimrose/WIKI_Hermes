# SKILL.md 작성·생애주기 원칙

[[Agent_Skill_Orchestrator_3요소]]의 *Skill* 요소를 어떻게 쓰고 언제 버릴지에 대한 실천 원칙. 스킬의 *내용*이 아니라 *작성 방식·로딩 구조·은퇴 시점*을 다룬다 — Agent Skills 표준(SKILL.md + scripts/references/assets)의 작성 모범에서 추출.

## 1. 설명이 곧 트리거다

스킬의 `description`은 단순 메타데이터가 아니라 *발동 조건* 그 자체다. 에이전트는 설명을 보고 "지금 이 스킬을 부를까"를 판단하므로, 설명의 정밀도가 트리거 정확도를 직접 좌우한다.

```
❌ "문서 작업 보조"                                       (모호 → 오발/미발)
✅ ".docx 생성·편집·분석; 변경 추적·주석·서식·텍스트 추출 처리"   (구체 → 정확)
```

> 출처 문서는 "설명 개선만으로 ~50% 성능 향상"을 주장하나 이는 *수치 미검증*이다. 경향(설명 정밀도 ↑ → 트리거 정확도 ↑)만 취한다.

이는 [[구체지시_우선_원칙]]의 SKILL 메타데이터 판본이다 — 구체적 설명일수록 강하게 발동한다.

## 2. 점진적 로딩 (Progressive Loading)

스킬 정보는 *한 번에 다 싣지 않고 단계적으로* 로딩된다. [[4계층_메모리_아키텍처]]가 메모리를 시간 척도로 계층화하듯, 스킬은 *필요 시점*으로 계층화한다.

| 계층 | 로딩 시점 | 권장 |
|---|---|---|
| frontmatter (name·description) | *항상* 상주 | 최소화 — 상시 토큰 비용 |
| SKILL.md 본문 | 트리거 시 | ≤500줄 |
| references/ | 필요 시 온디맨드 | 멀티토픽은 분리 |

frontmatter만 상시 비용이라는 점이 [[LLM_통제_철학_6대원칙]] 원칙 1(토큰 1급)과 직결된다 — 스킬을 늘릴수록 frontmatter가 누적되어 베이스라인을 잠식한다(같은 노트 "도구 스키마도 토큰이다"와 동일한 압력).

## 3. 절차가 아니라 결과를 기술하라

스킬 본문은 *단계 순서*가 아니라 *원하는 결과*를 기술한다. 세밀한 단계 시퀀스는 스킬이 아니라 scripts/로 내린다.

```
❌ "config 읽기 → DB URL 찾기 → 포트 수정"
✅ "사용자가 지정한 값으로 DB 포트를 갱신한다"
```

이유: 결과 기술은 모델이 *일반화*해 다양한 상황에 적용하게 하지만, 고정된 절차는 예외에서 깨진다. 결정론이 필요한 절차는 코드(script)로 박는다 — [[하네스_엔지니어링]]·[[Karpathy_4대_코딩원칙]]의 "결정론은 코드로, 유연성은 자연어로" 분업과 같다.

## 4. 부정 케이스를 명시하라

*언제 쓰지 말아야 하는가*를 설명에 적는다. 발동 조건만큼 비발동 조건이 오발을 막는다.

```
✅ "PDF 작업에 사용; 일반 문서·스프레드시트·평문엔 적용 금지"
```

[[Sonar_Protocol]]이 허용 대안과 *금지 명령*을 함께 적는 것과 같은 정신 — 경계를 양쪽에서 그어야 행동이 통제된다.

## 5. 은퇴 시점을 인지하라 (Capability Skill 한정)

에이전트가 *스킬 없이도* 평가를 통과하기 시작하면, 그 스킬의 가치는 이미 모델이 흡수한 것이다 — 폐기한다. 모델이 발전할수록 [[Agent_Skill_Orchestrator_3요소]]의 *Capability Skill*(능력 보강형)이 이 운명을 맞는다.

단 *Preference Skill*(팀 워크플로 준수형)은 예외다. 팀 규약인 한 모델 발전과 무관하게 유지된다 — 은퇴 원칙을 Preference Skill에 잘못 적용하면 워크플로가 무너진다.
