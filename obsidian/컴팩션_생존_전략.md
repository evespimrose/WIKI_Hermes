# 컴팩션 생존 전략

[[LLM_컨텍스트_5대_문제]] 1번(컴팩션 절벽)에 대한 구조적 응답. 컴팩션의 *전·후·시작* 세 시점에 각각 자동 훅을 걸어 핵심 상태가 사라지지 않도록 한다. [[LLM_통제_철학_6대원칙]]의 "Hooks > Prompts"가 가장 두드러지게 드러나는 구현 영역이다.

## 세 시점의 훅

### 1. 컴팩션 직전 — pre-compact.sh

자동으로 백업 파일을 작성한다:

```
.claude/memory-bank/pre_compact_backup.md
  ─ RIPER State 사본
  ─ 현재 플랜 파일 전문
  ─ 사용자 지시 메모
```

원리: 컴팩션이 일어나기 직전, 사라질 위험이 가장 큰 휘발성 상태들을 [[4계층_메모리_아키텍처]] L2·L3 영역으로 이주시킨다.

### 2. 컴팩션 직후 — post-compact.sh

자동으로 재주입한다:

```
CORE_RULES 6개 (RULE-1 ~ RULE-6)
복원된 RIPER State
Pre-Compact 백업
즉시 수행 지침: /memory:recall → 플랜 파일 로드
```

원리: 컴팩션 직후 AI는 "방금 무엇을 하고 있었는지" 모르는 상태가 된다. 즉시 백업 + 핵심 규칙을 재주입해 작업 연속성을 강제 복원한다.

### 3. 매 세션 시작 — session-start.sh

```
CORE_RULES + CLAUDE.md + .riper-state + 최신 플랜 + 최신 cxt 2개
```

원리: [[4계층_메모리_아키텍처]] L1(세션 컨텍스트)을 매번 새로 채우는 자동 장치. 사용자가 *AI가 무엇을 잊었는지* 추적할 필요가 없도록 보장한다.

## 왜 프롬프트가 아니라 훅인가

같은 일을 "컴팩션 전에 메모리 저장하세요"라는 프롬프트로 요구할 수 있지만, 컴팩션은 정의상 *그 프롬프트가 사라지는 순간*에 일어난다. 프롬프트로는 자기 자신의 소멸을 방어할 수 없다. 결정론적 외부 강제만이 이 문제를 푼다.

## 백업의 단방향성

훅들은 *AI가 아니라 시스템이* 작성한다. AI는 백업의 내용에 개입하지 않으며, 백업 파일을 사용자가 검증하지 않는 한 AI는 그 백업이 "무엇을 담고 있는지" 추측에 의존한다. 이 단방향성이 [[LLM_통제_철학_6대원칙]]의 "Single Responsibility per Role"과 결합해 환각 누적을 추가로 막는다.

## 자동 압축의 구조적 한계

왜 자동 `/compact`만으로 부족하고 *수동 개입*이 필요한가에 대한 근거. 자동 요약은 *이미 성능이 저하된 모델*이 자신의 컨텍스트를 압축하는 작업이다 — Dumb Zone([[LLM_컨텍스트_5대_문제]])에 들어선 모델이 "무엇이 중요한지"를 판단해 요약하므로, 정작 살려야 할 결정 근거·진행 단계가 누락될 수 있다. 압축의 품질은 압축을 수행하는 시점의 모델 상태에 종속되고, 그 시점은 정의상 가장 나쁜 상태다.

그래서 본 볼트는 *수동 스펙 문서·[[doc_context_입력_워크플로우]] 주입*을 채택한다 — 자동 요약에만 맡기지 않고, 사람(또는 성능이 온전한 시점의 에이전트)이 핵심을 미리 외부화한다. pre-compact 백업이 *시스템*이 쓰는 안전망이라면, 수동 스펙은 *사람의 판단*이 들어간 상위 신호다.

## 수동 스펙 Breadcrumb — 자동화 설계의 수동 프로토타입

세션을 넘길 때 *사람이 직접 중요도를 판단해* 다음 세션이 따라올 빵부스러기(breadcrumb) 스펙을 남기는 접근법. 자동 요약과 달리 *무엇이 핵심인지*를 온전한 판단력으로 선별해 전달하므로 신호 밀도가 높다.

이것은 본 볼트가 지향하는 *자동 키워드 추출*(`vault_mandate` 갭분석 Priority 1 "압축 키워드 자동 추출 규약")의 **수동 프로토타입**이다 — 자동화를 설계하기 전에, 사람이 손으로 하는 버전이 무엇을 남기고 무엇을 버리는지가 자동화의 기준 모델이 된다. 수동 breadcrumb이 검증한 "남길 가치"의 판단 로직을, 추후 [[위키_저장_5대필터]]·압축 규약으로 기계화하는 것이 로드맵 방향이다.
