# 작업 규모별 워크플로

[[LLM_통제_철학_6대원칙]] 6번(Scope-Differentiated Gates)의 직접 구현. 모든 작업에 동일 절차를 적용하지 않고, *규모에 따라* 게이트 깊이와 [[RIPER_5단계_상태머신]] 사용 여부를 다르게 한다.

## 규모 판단의 단일 기준

**플랜 파일 존재 여부**가 유일한 기준이다. `.claude/memory-bank/{branch}/plans/`에 해당 작업의 플랜이 있으면 중·대형, 없으면 소형.

이 기준이 단순한 이유: AI가 매번 "이건 큰 작업인가 작은 작업인가"를 추론하면 환각이 생긴다. 객관적 파일 존재 여부로 분기시키면 추론이 사라진다.

## 세 가지 워크플로

### 소형 — 버그픽스·단일 함수 수정

```
수정
  → quality-sentinel (코드 컨벤션 검증만 — RIPER 감사 생략)
  → reporter (work.md 기록, 문서 동기화)
```

특징:
- RIPER strict 모드 금지 (토큰 낭비)
- producer 경유 안 함
- quality-sentinel은 Phase 2(컨벤션)만 실행, Phase 1(RIPER 감사) 생략

### 중형 — 피처 개발·시스템 추가

```
/riper:strict → /riper:research → /riper:plan (명세 저장)
  → /riper:execute
      ↳ [구현 완료마다] quality-sentinel → reporter
  → /riper:review
  → producer conclusion.md
```

특징:
- 풀 RIPER 사이클
- quality-sentinel Phase 1+2 모두 실행
- producer가 conclusion 작성

### 대형 — 시스템 리팩토링·멀티 서브시스템

```
/riper:strict → /riper:research → /riper:innovate → /riper:plan
  → /riper:execute (서브에이전트 위임)
      ↳ [각 서브에이전트 완료마다] quality-sentinel → reporter
  → /riper:review
  → producer conclusion.md
```

차이점: INNOVATE 단계 추가, 서브에이전트 위임 패턴 사용.

## EXECUTE 진입 전 준비도 5축 체크

중·대형의 PLAN→EXECUTE 전이 직전, *착수 준비가 됐는지*를 5개 축으로 점검한다. 외부 프레임워크의 Confidence Gate에서 **점수화는 버리고 체크 축만** 차용한 것 — LLM이 자가 점수를 부풀릴 수 있어([[LLM_통제_철학_6대원칙]] 원칙3 *훅 > 프롬프트*와 긴장) *점수*가 아니라 *훅 체크리스트*로 구현한다.

| 축 | 묻는 것 |
|---|---|
| 중복 | 이미 있는 것을 다시 만들고 있지 않은가? |
| 아키텍처 | 기존 구조·경계와 정합하는가? |
| 공식 문서 | 근거가 되는 공식 문서를 확인했는가? |
| 선례(OSS) | 검증된 외부 구현·선례가 있는가? |
| 근본 원인 | 증상이 아니라 근본 원인을 짚었는가? ([[루트_원인_우선_디버깅]]) |

5축을 모두 통과해야 EXECUTE에 진입한다. 점수 임계(예: ≥90 진행)는 신뢰하지 않는다 — 원천 프레임워크조차 출시 전까지 5항목 중 4개가 스텁이었던 *표본 8개의 과대 주장*이었다. 신뢰할 것은 *항목 자체*이지 점수가 아니다. 구현은 [[L1_Hard_Gate_훅체계]]처럼 결정론적 훅으로 박아야 텍스트 규칙의 attention-drop을 면한다.

## 게이트 적용의 차등

| 규모 | RIPER 감사 | 코드 컨벤션 | reporter | producer conclusion |
|---|---|---|---|---|
| 소형 | 생략 | ✓ | ✓ (기본 모드) | 생략 |
| 중형 | ✓ | ✓ | ✓ (확장 모드) | ✓ |
| 대형 | ✓ | ✓ | ✓ (확장 모드) | ✓ |

게이트 에이전트들의 자세한 책임은 [[Producer_QualitySentinel_Reporter_게이트]]에서 다룬다.

## 왜 차등이 필요한가

[[LLM_컨텍스트_5대_문제]] 3번(토큰 폭발)을 막기 위한 구조다. "버그픽스 한 줄 수정"에 RIPER 풀사이클을 강제하면 *플랜 작성·승인·리뷰* 자체가 수정 비용의 수십 배가 된다. 절차의 비용은 절차의 가치를 초과해서는 안 된다.

같은 원리가 [[Producer_QualitySentinel_Reporter_게이트]]의 quality-sentinel "소형 작업에 RIPER 위반 판정 금지" 제약과 reporter "단순 조회 작업 개입 금지" 제약에 동일하게 적용된다.
