# 루트 원인 우선 디버깅 (Root-Cause-First)

증상에 즉흥적으로 패치를 붙이기 전에 *근본 원인을 먼저 규명*하도록 강제하는 4단계 디버깅 프로토콜. [[RIPER_5단계_상태머신]]의 RESEARCH 단계를 *디버깅에 특화*시킨 형태로, 볼트에 부재했던 디버깅 방법론을 채운다.

## 4단계

```
1. 조사 (Investigation) — 증상·재현 조건·관련 코드 경로 수집
2. 패턴 분석 (Pattern)    — 유사 사례·공통 구조 식별
3. 가설 (Hypothesis)      — 근본 원인 가설 수립·검증 방법 정의
4. 구현 (Implementation)  — 가설이 검증된 뒤에만 수정 착수
```

흐름은 *선형 의무*다. 1~3을 건너뛰고 4로 점프하는 것이 가장 흔한 실패 — 증상만 덮은 수정은 다른 곳에서 재발한다.

## 3회 실패 → 아키텍처 에스컬레이션

같은 버그에 *3회 이상 수정이 실패*하면, 그것은 국소 버그가 아니라 *아키텍처 문제*의 신호다. 이때는 같은 레이어에서 더 시도하지 말고 상위(설계) 문제로 끌어올린다. 이 규칙은 [[LLM_통제_철학_6대원칙]] 원칙 6(소형/중대형 구분)과 결합해 에이전트의 *무한 수정 루프*를 끊는 장치가 된다.

## 오버킬 방지 — 규모 게이팅

4단계 전체를 *모든 버그*에 강제하면 한 줄 오타에도 과한 절차가 붙는다. [[작업_규모별_워크플로]]의 "플랜 파일 존재 = 중·대형" 기준으로 디버깅 규모를 게이팅해, 소형 버그는 직접 수정하고 중·대형에만 4단계를 적용한다. 절차의 비용이 절차의 가치를 넘지 않도록.
