# L1~L5 다층 방어선의 역사

[[Sonar_Protocol]] 위반을 막기 위해 한때 운영됐던 5중 방어선과, 그 후 2026-05-27 최적화로 L1 + L3만 잔존하게 된 경위. [[Workflow_진화_3단계]] 2단계와 3단계 사이의 절충 과정.

## 5중 방어선 구조 (제거 전)

```
사용자 입력 → L5 → L4 → L3 → L2 → L1 → 도구 실행
```

| 계층 | 트리거 | 역할 | 운명 |
|---|---|---|---|
| L1 Hard Gate | PreToolUse | `cave-man-guard.sh`, `codegraph-gate.sh`, PowerShell guard로 도구 차단 ([[L1_Hard_Gate_훅체계]]) | **유지** |
| L2 Auto Route | PreToolUse | `auto-route-reminder.sh` — codegraph 권장 안내 | 제거 (L1 shadow) |
| L3 SessionStart | SessionStart | `session-start.sh` CORE_RULES 강제 재주입 | **유지(압축)** |
| L4 Audit Footer | Stop | `audit-footer.sh` — 턴 종료 후 사후 통지 | 제거 (L1 shadow) |
| L5 Meta Reminder | UserPromptSubmit | `meta-reminder.sh` — 매 turn critical reminder | 외부 이전 (context-sharer) |

## Shadow 관계의 발견

후속 분석에서 다음이 드러났다:

- **L1이 즉시 차단**하므로 L2가 "권장 안내"해도 의미 없음 (이미 차단됨)
- **L1이 즉시 차단**하므로 L4가 "사후 통지"해도 의미 없음 (실행이 안 됨)
- **L5는 매 턴 +150 토큰**을 쓰지만 attention drop으로 효과 ~50%

L2/L4/L5는 L1과 *기능 중복*(shadow) 관계였고, 매 턴 +517 토큰의 누수만 일으키고 있었다. 텍스트 reminder의 효과 한계는 [[LLM_통제_철학_6대원칙]] 원칙 3(훅 > 프롬프트)의 정량적 재확인이기도 하다.

## 절충: 시나리오 X

절충 검토 끝에 다음 시나리오가 채택됐다:

```
유지:   L1 (Hard Gate) + L3 (SessionStart 압축)
제거:   L2, L4
외부 이전: L5 → context-sharer SKILL.md (cxt 파일 3행 임베드)
```

### 트레이드오프

| 지표 | Before | After | 변화 |
|---|---|---|---|
| 턴당 고정 토큰 | +827 | +310 | **-62%** |
| 정책 파일 수 | 11 | 5 | -55% |
| 보장률 | 99.2% | 94% | -5.2%p |

5%p의 보장률 손실을 토큰 62% 절감으로 교환했다. 손실분은 context-sharer로 부분 보완된다. 네 개의 후보 시나리오 중 X가 채택된 의사결정 과정은 [[L1_방어선_시나리오X_결정]]에서 별도로 다룬다.

## L5의 외부 이전: context-sharer

trae(외부 AI)의 `context-sharer` 스킬이 cxt 파일 생성 시 **3행에 CAVE-MAN-REMINDER를 의무 임베드**한다:

```markdown
# [제목]
<!-- BLK: BLK-XXX -->
<!-- CAVE-MAN-REMINDER: codegraph 우선 (codegraph_context → search → node).
     find/grep -r/ls -r/rg/fd/cat .cs 자동 차단됨 -->

## Task
...
```

매 턴 reminder를 hook으로 강제하던 것을 *cxt 파일이 트리거 시점에 1회*로 변환한 것. 효과는 cxt 파일을 입력으로 받는 모든 작업에 한정되지만, 그게 RX_1 워크플로우의 거의 전부다.

이는 [[External_AI_역할분리]]가 정의한 외부 AI 권한 모델 안에서 trae가 *cxt 파일 작성*이라는 단일 책임으로 reminder 임베드를 떠맡는 것이라 책임 분리 원칙 위반이 아니다.

## 제거된 파일 4종

```
.claude/hooks/auto-route-reminder.sh   (L2)
.claude/hooks/audit-footer.sh          (L4)
.claude/hooks/meta-reminder.sh         (L5)
.claude/rules/cave-man-protocol.md     (CLAUDE.md에 통합)
```

`cave-man-protocol.md`는 별도 파일에서 CLAUDE.md 내 압축 규칙으로 흡수됐다.

## 잔존 정책 구조 (현재)

3개 핵심 L1 hook + 인프라 3종 (자세한 매트릭스는 [[L1_Hard_Gate_훅체계]]에서).

```
L1 Hard Gate 트리오:
  cave-man-guard.sh
  cave-man-guard-powershell.sh
  codegraph-gate.sh

인프라:
  tool-history-recorder.sh      (L1 검증 기반)
  write-approval-reminder.sh    (Write/Edit 보호)
  session-start.sh              (L3, CORE_RULES 압축)
```

## 롤백 보존

본 정책의 *제거된 상태도 보존*된다 — 미래에 보장률을 99.2%로 복원해야 할 가능성을 위해. 롤백 시:

1. 위키의 본 노트와 [[Workflow_Ecosystem_버전계보]] v3.4 항목을 참조해 절차 재구성
2. `meta-reminder.sh`, `audit-footer.sh`, `auto-route-reminder.sh`를 재생성
3. settings.json에 UserPromptSubmit, Stop 이벤트 재등록

이 보존은 [[LLM_Wiki_운영체계]]의 archive 원칙(원본 보존)과 같은 정신이다 — 결정이 바뀌어도 과거의 결정 근거는 사라지지 않는다.

### 5중 방어선 운영 당시 변경 파일 (롤백 시 참고)

L1~L5 전체 운영 직전의 변경 인벤토리. 롤백 시 이 목록의 신규 파일을 재생성해야 한다.

**신규 생성 (8개)**:
```
.claude/hooks/codegraph-gate.sh            (L1 - 유지)
.claude/hooks/cave-man-guard-powershell.sh (L1 - 유지)
.claude/hooks/tool-history-recorder.sh     (인프라 - 유지)
.claude/hooks/auto-route-reminder.sh       (L2 - 제거됨)
.claude/hooks/audit-footer.sh              (L4 - 제거됨)
.claude/hooks/meta-reminder.sh             (L5 - 외부 이전됨)
.claude/state/tool-history.log             (상태 파일)
.claude/state/violation-count.log          (상태 파일)
```

**수정 (5개)**:
```
.claude/settings.json                  (이벤트 등록)
.claude/hooks/session-start.sh         (L3 위반 통계)
.claude/hooks/cave-man-guard.sh        (cat/head/tail .cs 차단 강화)
.claude/hooks/write-approval-reminder.sh (codegraph_impact 검증 추가)
CLAUDE.md                              (L1~L5 명시)
```

이 인벤토리는 [[Workflow_Ecosystem_버전계보]] v3.4 항목과 짝을 이룬다 — 한쪽이 *왜 바꿨나*, 한쪽이 *무엇을 바꿨나*를 답한다.

## 후속 모니터링 임계치

최적화 적용 후 1주일 관찰 임계치:

- 위반 7일 누적 > 20회 → L5 복구 검토
- 토큰 절감 < 50% → 추가 최적화 검토
- 사용자 개입 +30% 이상 → context-sharer 임베드 강화
