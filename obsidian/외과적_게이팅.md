# 외과적 게이팅 (Surgical Gating)

차단 규칙은 *대체재가 존재하는 행위에만* 외과적으로 적용해야 한다는 원칙. 정당한 사용까지 막는 차단은 우회 비용으로 되돌아와 본말전도가 된다. [[L1_Hard_Gate_훅체계]]가 Grep을 *전면* 차단하지 않고 *소스 필터가 걸린 Grep만* 차단하도록 설계된 근거.

## 문제: Grep의 이중 용도

Grep 도구는 성격이 다른 두 작업에 쓰인다:

| 용도 | 예시 | codegraph 대체 |
|---|---|---|
| 구조적 검색 | "PlayerController 클래스 어디?" | ✅ [[Codegraph_MCP_통합]]이 더 정확·저렴 |
| 리터럴 텍스트 검색 | 에러 메시지 문자열, 주석, 로그 포맷 | ❌ codegraph 인덱스에 없음 |

구조적 검색은 codegraph로 대체하면 토큰이 줄지만, 리터럴 검색은 *대체재가 없다*. codegraph 인덱스는 심볼 그래프라 문자열 리터럴을 담지 않기 때문이다 — 이 경계는 [[First_Grep_실전사례]]에서 "텍스트 질문은 grep이 정답"으로 이미 다뤄졌다.

## 전면 차단의 역효과

Grep을 무조건 막으면 리터럴 검색까지 막혀, 모델이 *불필요한 codegraph 호출*로 우회한다. 사전 분석상 약 **+52% 토큰 역효과** — 비용을 줄이려던 규칙이 오히려 비용을 늘린다.

```
차단 목적:   토큰 절감
잘못된 차단: 정당한 사용까지 차단 → 우회 비용 발생 → 총비용 증가
올바른 차단: 대체재 있는 행위만 → 절감만 남고 역효과 없음
```

## 해결: 소스 타입 필터 분기

`codegraph-gate.sh`는 Grep의 `type`(cs/py/ts…) 또는 `glob`(*.cs…) 파라미터를 검사한다:

- 소스 코드를 가리키는 필터 + codegraph 미사용 → `ask`로 차단 (구조적 검색으로 판정)
- 필터 없는 검색 → 통과 (리터럴 검색으로 판정)

행위의 *의도*를 파라미터로 추론해 외과적으로 가른다.

## 차단은 길 안내를 동반한다

막기만 하면 모델이 헤맨다. 차단 메시지에 올바른 대안을 넣는다:

> "리터럴 텍스트 검색이면 type/glob 소스 필터를 빼면 통과됩니다."

이는 [[Sonar_Protocol]]이 금지 명령과 허용 대안을 *항상 쌍으로* 제시하는 패턴과 같다. 차단은 처벌이 아니라 *경로 재안내*다.

## 일반 원리

> 차단 규칙은 비용을 줄이려고 만든다. 그 규칙이 정당한 사용을 막아 우회 비용을 유발하면 본말전도다. 대체재가 있는 행위에만 외과적으로 적용하고, 막을 때는 올바른 길을 함께 안내한다.

이 원리는 [[이중방어_Defense_in_Depth]]의 hard 층을 *어떻게 정교하게 깎을 것인가*에 대한 답이다. 이중 방어가 "두 층을 깔라"면, 외과적 게이팅은 "각 층을 과잉 차단하지 마라"다.
