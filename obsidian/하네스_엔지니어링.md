# 하네스 엔지니어링 (Harness Engineering)

"AI가 일하는 *시스템(환경)*을 설계하는 것" — LLM 모델을 제외하고 그것을 둘러싼 에이전트·스킬·오케스트레이터·도구·메모리 전체를 가리킨다. **이 볼트 전체 철학의 외부 학술 명명**이다: [[Mastermind_Architecture_Manifesto]]의 "기계식 통제 섀시"가 곧 harness다.

## 어원과 명제

harness = 마구(말 다루는 도구). *말(AI)을 어떻게 다룰 것인가*. 핵심 명제: **같은 모델도 하네스 구성에 따라 생산성·품질이 크게 달라진다.** 모델 성능은 이미 충분해, 현재 병목은 모델이 아니라 *모델을 감싸는 구조*다 — [[LLM_워크플로우_생태계]] 전체가 이 전제 위에 서 있다.

## 3대 핵심 구성요소

[[Agent_Skill_Orchestrator_3요소]]가 본체. 요약하면 Agent(수행 주체)·Skill(단일책임 도구)·Orchestrator(PM). 본 볼트의 [[3Tier_에이전트_스튜디오]]·[[Producer_QualitySentinel_Reporter_게이트]]·스킬 생태계가 정확히 이 3요소다.

## 멀티에이전트 우위 (정량 근거)

단일 에이전트보다 멀티에이전트가 일을 더 잘한다(주방 비유: 1인 셰프는 주문 늘면 병목). 정량 평가: 15개 개발 과제를 하네스 유무로 비교, **평균 60% 성능 향상·100% 승률**, 과제가 어려울수록 단일 AI는 점수 하락하나 하네스는 품질 유지. 트레이드오프는 *토큰 +30%*([[작업_규모별_워크플로]]로 대형에만 적용해 상쇄).

## 검증 루프와 이중 검증

하네스는 한 번에 완성품을 내지 않는다 — 결과는 초안, 에이전트를 깎으며 수렴. 입력 신선도·중간산출·최종물을 피드백 루프로 검토. 에이전트별 다른 모델 연결도 가능: QA를 Codex(GPT-5.5)로 2차 검증 = 클로드 1차 + 코덱스 2차 *이중 검증*([[External_AI_역할분리]]의 도구 분리와 정합). 출처 기반 작성이라 할루시네이션 대폭 감소.

## 메타 하네스 — ready-to-order

목표(의도)만 주면 하네스를 *자동 구성*하는 메타 스킬(revfactory/harness)이 존재한다. "X 작성하고자 합니다"(의도) → 에이전트팀+스킬+오케스트레이터 6단계 자동 생성. 본 볼트 워크플로우와의 통합 타당성은 `docs/Try_HarnessWorkflowIntegration`에서 분석 — 공장(harness) ⊂ 헌법(RIPER) 레이어드.
