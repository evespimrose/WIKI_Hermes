# Try Skill — 타당성 검증

[[RIPER_5단계_상태머신]] 진입 *이전 단계*에 위치하는 사전 타당성 분석 스킬. 코드를 건드리기 전 "이 변경이 실제로 가치 있는가"를 5개 비교 축으로 검증한다. [[LLM_통제_철학_6대원칙]] 1번(Tokens as First-Class)과 [[Mastermind_Architecture_Manifesto]]의 카파시 정신(Surgical Changes)이 결합된 결과물.

## 위치

```
.claude/skills/try/SKILL.md
```

## 5개 비교 축 (하나라도 생략 금지)

| 축 | 내용 |
|---|---|
| **아키텍처 명확성** | 레이어 경계·의존성 방향·순환 참조 위험 |
| **컴파일·빌드 영향** | 증분 컴파일 속도·asmdef 수·참조 그래프 |
| **장기 유지보수** | 온보딩·레이어 위반 조기 감지·테스트 격리 |
| **단기 비용** | 마이그레이션 범위·씬/prefab 재설정·namespace 파급력 |
| **현재 작업과의 연관성** | 미결 사항·플랜과의 연계·착수 최적 타이밍 |

다섯 축은 *모두 반드시* 평가된다. 한 축만 좋아 보여 진행하면 나머지 축에서 손해 보는 패턴(예: 단기 비용은 낮지만 장기 유지보수가 망가지는 변경)을 미리 잡는다.

## 출력 양식

```
docs/Try_{주제}.md

# Try — {제목}
## 1. 현황 (As-Is)
## 2. 제안 구조 (To-Be)
## 3. 장단점 비교 (3-A ~ 3-E: 5개 축)
## 4. 순환 의존성·방향 위험 분석
## 5. 종합 평가 및 추천
## 6. 미결 사항 연계 (해당 시)
```

각 Try 분석은 *별도 파일*로 영속화된다 — [[4계층_메모리_아키텍처]] L3(장기 기억) 계층에서의 의사결정 근거 자료.

## 착수 경계 — 사용자 어휘별 행동

Try 출력 후 사용자 응답에 따라 *명확히 다른 길*로 분기한다.

| 사용자 응답 | 행동 |
|---|---|
| "해줘" / "OK" | producer 경유 구현 착수 |
| "알겠어" / 침묵 | `docs/Try_xxx.md` 기록만, 대기 |
| "다른 옵션도" | 추가 try 수행 |

이 어휘 매핑은 *암묵적 동의*와 *명시적 결재*를 어휘 수준에서 분리한다. AI가 "알겠어"를 진행 신호로 오해하지 않게 만드는 [[LLM_컨텍스트_5대_문제]] 2번(환각) 방어.

## 왜 RIPER 전에 두는가

RIPER PLAN 단계는 *방향이 결정된 후*의 명세 작성이다. 방향 자체가 잘못된 변경은 PLAN을 잘 짜도 무의미하다. Try는 *방향 결정 단계*에 단독으로 위치해, RIPER가 시작되기 전에 "이 작업을 할 필요가 있는가"를 먼저 검증한다.

```
사용자 요청
   ↓
Try (5축 분석)
   ↓
"해줘" 응답
   ↓
RIPER 진입 (Research → ...)
```

## "Try"라는 어휘

"Plan" 대신 "Try"인 이유는 *시도해보는 단계*임을 명시하기 위함이다. 진지한 명세가 아니라 *얕은 비교 분석*이다. 짧고 가벼운 형태로 운영되어야 다섯 축 평가의 비용이 작업 비용을 초과하지 않는다.

## 탐색 규약 (Sonar Protocol)

try의 현황 파악(Stage 2)은 전체 작업 토큰의 가장 큰 비중(plain 기준 약 74%)을 차지한다. 여기서 [[Codegraph_MCP_통합]]을 쓰면 −85%까지 절감되지만, 탈옥하면 그 절감이 통째로 사라진다. 한때 try가 codegraph를 건너뛰고 파일을 광범위하게 읽는 탈옥이 잦았던 이유는 실행 프로토콜 1단계가 `READ 코드베이스 파악`이라 적혀 codegraph 언급이 없었기 때문이다 — 스킬이 스스로 "읽어도 된다"고 자가인가한 [[구체지시_우선_원칙]]의 전형적 사례.

수정된 프로토콜은 탐색을 두 단계로 분리한다:

```
0. EXPLORE  codegraph 우선 — context → search → impact → explore
1. READ     codegraph로 부족한 부분만 Read(offset+limit) 보완
```

이제 try는 codegraph-first를 *자기 자신의 지시*로 인식해 전역 규칙과의 충돌이 사라진다. 텍스트만으로는 또 어겨질 수 있으므로 [[L1_Hard_Gate_훅체계]]의 Grep 매처 확장이 hard 층에서 함께 강제한다 — soft+hard [[이중방어_Defense_in_Depth]]의 실제 적용 사례다.

## Trae에서의 Try

외부 AI(Trae)에서도 동일한 5축 분석을 수행할 수 있다 → `.trae/skills/try/SKILL.md` ([[External_AI_역할분리]] 카탈로그 참조). 단 *외부 AI가 직접 결정·구현*하는 것은 금지이므로, Trae Try는 결과 문서만 작성하고 Claude로 넘긴다.
