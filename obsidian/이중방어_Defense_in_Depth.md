# 이중 방어 (Defense in Depth)

규율을 강제할 때 *soft 지시(텍스트)* 와 *hard 강제(hook)* 두 층을 동시에 까는 설계 원칙. 한 층은 반드시 어겨질 수 있다는 전제에서 출발한다. [[L1_Hard_Gate_훅체계]]와 스킬 텍스트가 함께 [[Sonar_Protocol]]을 지키게 만든 구조에서 추출됐다.

## 왜 한 층으로는 부족한가

[[LLM_통제_철학_6대원칙]] 원칙 3(Hooks > Prompts)은 "프롬프트는 신뢰할 수 없으니 훅으로 강제하라"고 말한다. 그러나 역도 성립한다 — *훅도 매처 공백으로 새기 때문에* 텍스트 규약이 근본 원인을 함께 막아야 한다. 두 층은 서로의 빈틈을 메운다.

| 층 | 수단 | 막는 것 | 약점 |
|---|---|---|---|
| Soft | 스킬/커맨드 텍스트 | 근본 원인 (모델이 옳게 행동하도록) | 모델이 어길 수 있음 |
| Hard | PreToolUse hook | 행위 자체 (어겨도 도구 차단) | 매처에 없는 도구는 못 막음 |

## 사례: try 스킬 codegraph 탈옥

[[Try_Skill_타당성_검증]] 스킬이 현황 파악 단계에서 codegraph를 건너뛰고 파일을 광범위하게 읽는 탈옥이 잦았다. 원인이 두 층 모두에 있었다:

1. **Soft 공백** — try SKILL.md 1단계가 `READ 코드베이스 파악`이라 적혀 codegraph 언급이 없었다. 스킬이 스스로 "읽어도 된다"고 자가인가했다 ([[구체지시_우선_원칙]]).
2. **Hard 공백** — `codegraph-gate.sh`가 `Read|Glob` 매처로만 등록돼 Grep 도구가 게이트 밖이었다.

한 층만 고치면 다른 층으로 샌다. 그래서 *동시에* 막았다:

- Soft: SKILL.md에 `Stage 0 EXPLORE (codegraph-first)` 단계 신설
- Hard: 매처를 `Read|Glob|Grep`으로 확장 ([[L1_Hard_Gate_훅체계]])

## 일반 원리

> 차단을 설계할 때 "이 한 층이 뚫리면?"을 항상 가정한다. soft 층은 모델의 협조를 얻고, hard 층은 비협조를 강제한다. 둘 다 있어야 협조·비협조 양쪽이 닫힌다.

이는 보안 공학의 Defense in Depth를 LLM 워크플로우에 옮긴 것이다. [[L1_L5_다층방어선_역사]]의 5중 방어선도 같은 정신이었으나, 그쪽은 *같은 성격(텍스트 reminder)의 층을 중복*해 shadow 잉여를 낳았다. 이중 방어의 핵심은 *층의 개수*가 아니라 *성격이 다른 두 층*(soft+hard)의 직교성이다.

## 매처 합집합 원칙 (Hard 층의 하위 규칙)

hard 층(hook)은 *등록된 도구(matcher)에만* 작동한다. 같은 목적을 달성하는 우회 도구가 매처에 빠지면 그게 탈옥구가 된다.

```
막을 행위: 소스 코드 탐색
가능한 도구: Bash(find/grep -r) · PowerShell(-Recurse) · Read · Glob · Grep
→ 매처 = 이 모든 도구의 합집합이어야 한다
```

try 탈옥은 정확히 Grep이 합집합에서 빠져 있었기 때문이다. 한 행위를 막으려면 *그 행위가 가능한 모든 도구 경로*를 매처에 넣어야 한다.
