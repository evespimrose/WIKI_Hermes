# LLM 컨텍스트 5대 문제

장기 LLM 운용에서 반복적으로 관찰되는 다섯 가지 본질적 문제. [[LLM_워크플로우_생태계]]의 모든 설계 결정은 이 다섯 문제 중 하나 이상을 직접 겨냥한다.

## 1. 컴팩션 절벽 (Compaction Cliff)

장기 세션에서 컨텍스트가 한계에 도달하면 `/compact` 또는 자동 압축이 실행된다. 이때 현재 진행 단계, 사용자가 강조한 규칙, 진행 중 RIPER 모드, 직전 결정의 근거가 모두 소실된다. AI는 이전 컨텍스트가 무엇이었는지 모르는 채 작업을 이어가며, 사용자는 매번 처음부터 다시 설명한다.

대응: [[컴팩션_생존_전략]]과 [[4계층_메모리_아키텍처]]의 L2·L3 계층이 이 문제를 분담한다.

## 2. 환각 누적 (Hallucination Accumulation)

세션이 길어질수록 LLM은 다음과 같이 드리프트한다.

- 명시된 규칙을 "기억한다"고 주장하지만 실제로는 우회
- 사용자 의도와 다른 방향으로 작업 확장
- 존재하지 않는 파일·함수·API 호출
- 권한 없는 코드 변경

LLM의 "Lost in the Middle" 현상과 결합해 컨텍스트 중반 정보가 소실되면 환각은 기하급수적으로 증가한다.

대응: [[LLM_통제_철학_6대원칙]]의 "Hooks > Prompts", "Single Responsibility per Role"이 이 문제의 주된 방어선이다.

## 3. 토큰 비용 폭발 (Token Cost Explosion)

작은 수정을 위해 큰 파일을 통째로 Read하는 패턴이 만연한다.

- 20줄 edit을 위해 1,500줄 파일 전체 로드 → ~15,000 토큰 낭비
- bash `find` / `grep -r` 결과를 컨텍스트에 흡수 → 수천 토큰 낭비
- "확인용" 반복 Read → 매 턴 수천 토큰 누적

대응: [[Sonar_Protocol]]과 [[Dictionary_md_좌표_시스템]]의 BLK 좌표 인덱싱이 핵심 방어 수단이다. [[RULE7_PRD_Spatial_Mapping]]은 EXECUTE 단계에서 좌표 없는 행동을 차단한다.

## 4. 세션 단절 (Session Discontinuity)

새 세션을 열거나 다른 AI(Trae, Gemini)에게 작업을 넘길 때 컨텍스트가 0에서 시작한다. 사용자는 모든 배경을 다시 설명해야 하며, AI마다 동작 규칙이 달라 동일 결과를 보장할 수 없다.

대응: [[4계층_메모리_아키텍처]]의 L1·L4 계층, [[External_AI_역할분리]], 그리고 [[doc_context_입력_워크플로우]]가 함께 이 문제를 푼다.

## 5. 에이전트 디스패치 혼란 (Agent Dispatch Confusion)

어떤 작업에 어떤 에이전트를 호출할지 자체가 복잡한 결정이 된다.

- 단순 질문에 Opus 에이전트 호출 → 비용 낭비
- 멀티시스템 작업을 단일 에이전트가 처리 → 품질 저하
- 검증 단계 누락 → 결함 누적

대응: [[3Tier_에이전트_스튜디오]]의 Tier 분리와 [[Producer_QualitySentinel_Reporter_게이트]]의 게이트 체인이 이 문제를 구조화한다. [[작업_규모별_워크플로]]는 규모별 라우팅을 정의한다.

## 임계 구간: Smart Zone vs Dumb Zone

위 다섯 문제는 *컨텍스트가 일정 분량을 넘어서면* 동시에 악화된다. 이를 두 구간으로 부르면 실용적이다 — 컨텍스트가 비교적 적을 때는 모델이 또렷하게 동작하는 **Smart Zone**, 일정선(경험적으로 ~100k 토큰 부근)을 넘어서면 추론·지시 준수가 눈에 띄게 무너지는 **Dumb Zone**.

이 경계는 *규범이 아니라 경험적 기준값*이다 — 모델·태스크별 편차가 크므로 "100k"를 하드코딩하지 않는다. 다만 "어디쯤부터 Dumb Zone인가"를 의식하면 *세션 분할*과 *강제 컴팩션 트리거*([[컴팩션_생존_전략]])의 임계값 설계에 실용적 기준이 된다. 핵심은 컨텍스트를 Smart Zone 안에 *유지*하도록 설계하는 것 — 이것이 [[Sonar_Protocol]]·[[doc_context_입력_워크플로우]]가 진입량을 통제하는 이유다.

## 방해 정보와 오류율 — Context Rot 실증

3번(토큰 폭발)과 2번(환각)이 *왜* 컨텍스트 길이에 비례해 악화되는지의 메커니즘. Chroma Research(18개 LLM 대상)는 *비슷하지만 틀린 정보(distractor)*가 맥락에 섞일수록 오류율이 컨텍스트 길이에 비례해 *급증*함을 보였다 — 단순히 길어서가 아니라, 긴 맥락에 끼어든 방해 정보가 모델을 잘못된 답으로 끌어당긴다.

이것이 [[Sonar_Protocol]]의 "무차별 탐색 금지"가 *왜 옳은가*에 대한 외부 실증이다. `grep -r`·전체 파일 Read는 *관련 같지만 실제론 방해가 되는* 정보를 대량 주입해, 토큰만 쓰는 게 아니라 *오류율 자체*를 끌어올린다. 좌표 기반 접근(BLK·codegraph)이 진입량을 줄이는 것은 곧 distractor를 줄여 환각을 줄이는 일이다.

## 통찰

이 다섯 문제는 모두 "LLM이 모든 것을 기억하고, 자유롭게 탐색하고, 스스로 판단하면 잘 될 것이다"라는 가정의 실패를 보여준다. 5대 문제의 공통 해법은 *AI의 자율성을 제한하고, 상태와 인덱스를 외부에 두는 것*이다. 이 통찰이 [[LLM_통제_철학_6대원칙]]의 출발점이다.
