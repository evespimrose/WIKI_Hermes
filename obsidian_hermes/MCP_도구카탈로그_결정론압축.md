# MCP 도구카탈로그 결정론 압축

업스트림 MCP 서버를 감싸는 stdio 프록시가 `tools/list`·`prompts/list`·`resources/list` 응답의 `description` 필드를 **코드로 압축**한다(caveman-shrink). 모델 자기준수가 아니라 `compress()` 함수가 결정론적으로 prose를 줄인다 — 코드·URL·경로·식별자는 verbatim 보존. 다수 MCP를 연결할수록 도구 카탈로그가 *상시 입력 토큰*으로 깔리는데, 그 설명문을 무손실 경계로 깎는 입력측 절감 장치다.

## 토큰 절감의 제3축

볼트의 토큰 통제는 두 축이었다 — 입력 *접근*([[Sonar_Protocol]]: 무엇을 읽을지)과 출력 *표현*([[출력_표현_압축]]: 어떻게 말할지). 본 패턴은 셋째 축을 연다: **도구 카탈로그 입력**(어떤 도구 설명을 컨텍스트에 들일지). 세 축은 직교하므로 가산된다.

## 결정론이 핵심 — 코드가 압축한다

[[결정론_검증_게이트]]가 "비결정 LLM 산출물을 *코드가 검증*"이라면, 본 패턴은 "입력 prose를 *코드가 압축*"이다. 둘 다 [[LLM_통제_철학_6대원칙]]의 "Hooks > Prompts"를 입력/산출물 레벨로 구현한 형제다. caveman 출력 스킬의 약점(프롬프트 자기준수 → 장세션 drift)을 코드 압축이 구조적으로 제거한다 — 모델이 협조하든 말든 프록시는 항상 같은 결과를 낸다.

## 보수적 경계

v1은 의도적으로 좁다: **요청 본문**과 `tools/call` **응답 본문**은 무변조로 통과시킨다(다운스트림 파싱을 침묵 파손할 위험 회피). 압축은 *읽기용 메타데이터*(설명문)에만 가한다. 줄여도 안전한 범위 = 모델이 도구 *선택*에 쓰는 산문, 줄이면 위험한 범위 = 도구가 *주고받는* 데이터.

## 볼트에서의 위치

[[Token_허브]] 입력 슬롯의 신규 메커니즘. opt-in 미들웨어라 미도입 시 런타임 +0%, 운용 시 list 응답만 1회 −N%. 단 과압축의 의미 손실 가능성 때문에 [[압축효과_측정_정직성]](대조군 대비 측정)과 [[압축도구_표방실제_검증]](정확도 실증 요구)을 반드시 동반해야 한다 — "코드가 압축"이 곧 "무손실"은 아니다.
